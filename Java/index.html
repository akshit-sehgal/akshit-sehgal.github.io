Part I

Foundations

Chapter 1 — Hello, Java

Before you write a single line of code, it helps to understand what Java is, why it still matters in 2026, and how it fits into the broader programming landscape. This chapter is mostly conceptual — by the end, you'll know what you've signed up to learn and why the pieces fit together the way they do.

What is Java?

Java is a general-purpose, statically-typed, object-oriented programming language. It was created at Sun Microsystems in the early 1990s by James Gosling and a team originally working on consumer electronics. Today it is owned by Oracle and developed in the open through the OpenJDK project.

Let's unpack each of those adjectives, because they shape how you'll write Java code every day.

General-purpose means Java isn't restricted to any one domain. You can build web servers, mobile apps, command-line tools, big data pipelines, or desktop applications with it. Other languages are more specialized: R is general-purpose too but is mostly used for statistics; MATLAB is mostly for numerical computing. Java sits in the same broad-purpose bucket as Python, C#, and JavaScript.

Statically-typed means every variable has a type known at compile time. The compiler checks that you don't try to do anything inappropriate with that type — like calling a string method on a number. Compare:

// Java — statically typed:
int x = 5;
x = "hello";   // COMPILE ERROR — can't assign String to int

// Python — dynamically typed:
x = 5
x = "hello"   # fine, x's type just changed

Static typing catches a huge class of bugs before your code ever runs. The trade-off is more typing (in both senses — keystrokes and explicit type declarations). Modern Java softens this with features like var (Chapter 4) that let the compiler infer types from context.

Object-oriented means Java's primary unit of organization is the class. Almost everything you write lives inside a class. A class bundles together related data (fields) and behavior (methods), and you create objects from those classes to actually do work. Part III of this book is dedicated to mastering OOP.

"Write once, run anywhere"

Java's defining promise is portability. Your code compiles to bytecode — a platform-independent intermediate format — which the Java Virtual Machine (JVM) then executes on Windows, macOS, Linux, or anywhere else a JVM exists. You don't recompile for each platform.

This is fundamentally different from how languages like C or Rust work. Those compile directly to native machine code for one specific CPU architecture and operating system. A C binary compiled on macOS won't run on Windows.

The Java workflow looks like this:

  Hello.java         →    javac    →    Hello.class      →    java    →    Output
  (source code)           (compiler)    (bytecode, .class)      (JVM)        (your
                                                                              program
                                                                              runs)

The compiler javac reads your .java source and produces a .class file containing bytecode. The java command launches a JVM that loads and executes that bytecode. The same .class file will run on any platform with a compatible JVM.

Bytecode isn't human-readable, but it's not random either

If you're curious, run javap -c Hello.class on a compiled file. It disassembles the bytecode into a readable instruction listing — things like iconst_5 (push the integer 5 onto the stack) and iadd (pop two ints, push their sum). It's a small stack-based virtual machine.

Why learn Java?

There are a lot of programming languages. Why pick Java? Five concrete reasons:

1. Ubiquity in enterprise software.

Java is the workhorse of large-scale enterprise systems. Banks, insurance companies, telcos, government services, e-commerce platforms — these run on Java backends, often with Spring Boot. If you want a job at a Fortune 500 company writing backend code, Java is one of the safest bets you can make.

2. Android.

Despite Kotlin's rise, Java is still a first-class Android language and underlies much of the Android framework. Knowing Java unlocks billions of mobile devices.

3. The big data ecosystem.

Hadoop, Spark, Kafka, Cassandra, Elasticsearch — most of the major distributed data systems are written in Java or another JVM language (Scala, Kotlin). They run on the JVM, they expose Java APIs, and the people who maintain them work in Java daily.

4. Stability and tooling.

Java's tooling is exceptional. IntelliJ IDEA is widely considered the best IDE on any platform; Maven and Gradle are mature, battle-tested build systems; JUnit is the testing framework many other languages have copied. Profilers, debuggers, and monitoring tools for production JVMs are extremely well-developed.

5. Performance.

Modern JVMs (HotSpot, GraalVM) deliver performance comparable to native code for long-running server applications. The JVM watches your code run and optimizes hot paths on the fly — Just-In-Time (JIT) compilation. A long-running Java server often outperforms a naive C program because the JIT can specialize for the actual data patterns it sees.

The Java ecosystem at a glance

Three acronyms you'll see constantly. They're related but distinct:

JVM (Java Virtual Machine).

The runtime engine that actually executes Java bytecode. It's a specification — different vendors provide their own implementations. HotSpot (the reference implementation in OpenJDK) is what most people use. The JVM handles memory management (garbage collection), thread scheduling, JIT compilation, and the platform-specific glue that makes "write once, run anywhere" possible.

JRE (Java Runtime Environment).

The JVM bundled with the standard library — enough to run Java programs, but not to compile them. Historically distributed separately, but modern JDKs include everything, so you rarely install just the JRE anymore.

JDK (Java Development Kit).

Everything in the JRE plus the development tools: the compiler (javac), the debugger (jdb), the documentation generator (javadoc), and the interactive shell (jshell). This is what you install to write Java code.

A useful mental model: JDK contains JRE contains JVM. Each layer adds something on top of the one below it. As a developer, you only need to install the JDK.

JDK vendors: who makes Java?

Here's where it gets a bit weird. There are multiple distributions of the JDK, all built from the same OpenJDK source code, but packaged and supported by different organizations. They're functionally equivalent for most purposes; the differences are in licensing, support, and a few performance tweaks.

  • Oracle JDK — Oracle's official build. Free for development and many uses; production commercial use may require a paid subscription. Read the current Oracle license carefully before deploying.
  • Eclipse Temurin (formerly AdoptOpenJDK) — fully open-source, free for any use including commercial. Maintained by the Eclipse Adoptium project. A solid default choice and what we recommend for learning.
  • Amazon Corretto — Amazon's distribution. Free, with long-term support. Popular if you're on AWS.
  • Azul Zulu — popular in enterprise environments. Free community edition; paid options for enhanced support.
  • Microsoft Build of OpenJDK — Microsoft's distribution. Free, with good Windows integration.
  • GraalVM — Oracle's high-performance JDK. Notable for native image compilation, which turns Java programs into self-contained native executables with millisecond startup times. Worth exploring once you're comfortable with Java.
Which JDK should you install?

For learning, install Eclipse Temurin 21 (the current Long-Term Support release). It's free, widely adopted, and stable. We'll cover installation in Chapter 2. Don't agonize over the choice — for everything in this book, they're interchangeable.

A brief history of Java

Knowing roughly when features arrived helps you read older codebases and understand why some Java code looks dated. Until 2017, Java releases were years apart and added a lot at once. Since Java 9, there's been a new release every six months, with Long-Term Support (LTS) versions every two or three years.

  • 1995 — Java 1.0: first public release. Applets and the JVM debut. Most of the syntax you'll write today already existed.
  • 2004 — Java 5: generics, the enhanced for-loop, autoboxing, enums, varargs. A massive language upgrade that made daily code much cleaner.
  • 2014 — Java 8: lambdas, the Stream API, the modern Date/Time API. The turning point — Java became a viable functional-programming language.
  • 2017 — Java 9: the module system (rarely used in apps, important in the JDK), jshell, factory methods like List.of(...).
  • 2018 — Java 11 (LTS): a new long-term support baseline; many organizations migrated here from Java 8.
  • 2021 — Java 17 (LTS): sealed classes, pattern matching for instanceof, records (formally), text blocks.
  • 2023 — Java 21 (LTS): virtual threads, sequenced collections, record patterns, pattern matching for switch. This is the current recommended LTS.
  • 2024–2026 — Java 22 through 25: ongoing improvements to pattern matching, structured concurrency, scoped values, the Foreign Function & Memory API.

If a codebase you encounter uses raw types, anonymous inner classes for every callback, and verbose getter/setter boilerplate, it's probably pre-Java 8. If it uses lambdas and streams but no records or sealed types, it's likely Java 8–16. Modern Java code uses all the recent features and reads noticeably more concisely.

How Java code actually runs

Let's go one level deeper than the diagram above. When you run java Hello, several things happen:

  • The JVM starts up — allocating memory, initializing the garbage collector, and loading core classes from the standard library.
  • It finds the Hello.class file (the classpath tells it where to look) and loads it.
  • It verifies the bytecode is well-formed and safe — the verifier checks types, stack usage, and access rules.
  • It locates the main method and starts executing.
  • Bytecode is initially interpreted — read and executed one instruction at a time. This is slower than native code.
  • As methods get called repeatedly, the JIT compiler kicks in, translates the hot bytecode into optimized native machine code, and the JVM switches over to running the compiled version. This is where Java's performance comes from.

You don't need to understand all of this to write Java — most developers never think about it. But knowing that the JVM is a sophisticated runtime, not just a naive interpreter, helps explain why Java performance is what it is.

Exercises

Exercise 1.1

Explain in your own words the difference between the JDK, JRE, and JVM. Which one(s) do you need to write Java code, and which to merely run a Java program?

Solution to Exercise 1.1

The JVM is the runtime engine that executes Java bytecode. It handles memory management, JIT compilation, and providing a consistent execution environment across platforms.

The JRE is the JVM plus the standard libraries needed to run programs.

The JDK is the JRE plus development tools: the compiler, debugger, and other utilities.

To write Java, you need the JDK. To merely run a Java program, the JRE is enough — though in practice you'll install a JDK because modern JDKs include everything and the standalone JRE is no longer distributed by most vendors.

Exercise 1.3

What does "statically typed" mean? Give one advantage and one disadvantage compared to a dynamically-typed language like Python.

Solution to Exercise 1.3

Statically typed means every variable has a type known at compile time, and the compiler checks that operations on that variable are valid for its type.

Advantage: Many bugs are caught at compile time rather than runtime. The IDE can offer accurate autocomplete and refactoring because it knows the type of every expression. Performance is usually better because the runtime doesn't need to check types dynamically.

Disadvantage: More verbose — you have to write out type declarations (though var helps). Less flexibility for things like generic data containers or duck-typed APIs. Refactoring across types can be more work.

Exercise 1.4

Java is often described as "compiled" by some and "interpreted" by others. Which is it, and why is the distinction interesting?

Solution to Exercise 1.4

Both, in stages. Your source code is compiled by javac into bytecode. That bytecode is then loaded by the JVM, which initially interprets it instruction by instruction. As methods get hot, the JIT compiler translates the bytecode into native machine code for the actual CPU you're running on — so the same code starts as interpretation and graduates to compilation.

The distinction matters because it explains Java's performance profile: slower startup (interpretation kicks in first; JIT needs time to warm up) but excellent steady-state performance once the JIT has done its work. It's also why benchmarks of short-lived Java programs can be misleading — they don't run long enough for the JIT to make a difference.

Chapter 2 — Installing Java and Your Tools

Installing the JDK

We recommend Eclipse Temurin 21 (LTS). Go to adoptium.net and download the installer for your operating system. The site auto-detects your OS and shows the right download.

If your organization or course requires a different distribution (Oracle JDK, Amazon Corretto, etc.), that's fine — everything in this book works the same way. The differences are in licensing, not in the language.

On Windows

  • Download the .msi installer.
  • Run it. Accept the option to set JAVA_HOME and add Java to your PATH — these aren't strictly required, but they make life easier.
  • Restart your terminal so it picks up the updated environment variables.
  • If you're using PowerShell, you may need to launch a new window for changes to take effect.

If you skip the PATH option during installation, you can add it manually under System Properties → Advanced → Environment Variables. Add the JDK's bin directory to the PATH variable.

On macOS

  • Download the .pkg installer, or use Homebrew with brew install --cask temurin@21.
  • Verify with /usr/libexec/java_home -V — this lists all JDKs on your system.
  • If you have multiple JDKs installed, set JAVA_HOME to pick the right one.
# In your ~/.zshrc (or ~/.bash_profile on older macOS):
export JAVA_HOME=$(/usr/libexec/java_home -v 21)
export PATH=$JAVA_HOME/bin:$PATH

On Linux

# Ubuntu/Debian — using the Adoptium APT repository:
sudo apt update
sudo apt install temurin-21-jdk

# Or download the tarball from adoptium.net and extract:
wget https://github.com/adoptium/temurin21-binaries/releases/download/.../OpenJDK21U-jdk_x64_linux.tar.gz
tar -xzf OpenJDK21U-jdk_x64_linux.tar.gz
# Then set JAVA_HOME and PATH in ~/.bashrc:

# Verify package install:

Verifying the installation

Open a terminal and run:

java -version
javac -version

You should see something like:

openjdk version "21.0.2" 2024-01-16 LTS
OpenJDK Runtime Environment Temurin-21.0.2+13 (build 21.0.2+13-LTS)
OpenJDK 64-Bit Server VM Temurin-21.0.2+13 (build 21.0.2+13-LTS, mixed mode)

Both commands should succeed and show version 21 or higher. If you get command not found or similar, your PATH isn't set up correctly — go back and check the installation step for your OS.

Why two commands?

java is the runtime — it executes compiled bytecode. javac is the compiler — it turns .java source files into .class bytecode files. They're separate executables; both come with the JDK.

Meet jshell — Java's REPL

Java 9 introduced jshell, a Read-Eval-Print Loop. It lets you type Java expressions and statements one at a time and see the result immediately — no need to write a full class, save a file, or compile anything. It's perfect for learning, exploring APIs, and trying out small snippets.

Start it from your terminal:

$ jshell

|  Welcome to JShell -- Version 21.0.2
|  For an introduction type: /help intro

jshell> 2 + 3
$1 ==> 5

jshell> String name = "Java"
name ==> "Java"

jshell> System.out.println("Hello, " + name)
Hello, Java

jshell> /exit

A few things to notice: you don't need semicolons inside jshell for single-line statements; you don't need to wrap things in a class; and unbound expressions get auto-named ($1, $2, etc.) so you can reference them later. Throughout the book, jshell is great for testing snippets without setting up a full project.

More jshell examples

Defining a method:

jshell> int square(int n) {
   ...>     return n * n;
   ...> }
|  created method square(int)

jshell> square(7)
$3 ==> 49

jshell> square(square(3))
$4 ==> 81

Defining a class:

jshell> class Point {
   ...>     int x, y;
   ...>     Point(int x, int y) { this.x = x; this.y = y; }
   ...>     public String toString() { return "(" + x + ", " + y + ")"; }
   ...> }
|  created class Point

jshell> Point p = new Point(3, 4)
p ==> (3, 4)

jshell> p.x

Useful jshell commands

  • /vars — list variables you've defined
  • /methods — list methods you've defined
  • /types — list classes and interfaces you've defined
  • /imports — show active imports (jshell auto-imports common packages)
  • /list — show the source of everything you've defined
  • /edit — open an editor to modify a snippet
  • /save file.txt — save your session to a file
  • /open file.txt — load a previously saved session
  • /exit — leave jshell
  • /help — show all commands

When NOT to use jshell

jshell is great for exploring and learning, but it's not a real development environment. For anything beyond quick experiments — writing tests, building applications, working with multiple files — use your IDE. Many Java developers use jshell occasionally; few use it daily.

Chapter 3 — Your First Java Program

In this chapter you'll write, compile, and run your first Java program from scratch — first using the command line, then using your IDE. We'll dissect every line and every keyword so the structure makes sense and isn't just magical incantation.

Hello, World

Create a file called Hello.java with this exact content. Use any text editor — Notepad, VS Code, IntelliJ, anything that saves plain text.

public class Hello {
    public static void main(String[] args) {
        System.out.println("Hello, World!");
    }
}

Now compile and run it from the terminal:

$ javac Hello.java     # produces Hello.class in the same directory
$ java Hello           # runs the program
Hello, World!

Notice: you compile by file name (javac Hello.java) but run by class name (java Hello, no .class extension). That trips up beginners regularly.

Line-by-line breakdown

Five lines, three of them just braces. Let's go through every meaningful piece.

public class Hello {

This line declares a class named Hello. Java is object-oriented; virtually all code lives inside a class. Even a one-line "hello world" needs to be inside a class.

The keyword public is an access modifier. It means this class is accessible from anywhere — any other class, any package. The alternative (omitting public) would make the class package-private — only visible to other code in the same package. We'll cover access modifiers thoroughly in Chapter 14.

The class name must match the file name exactly when the class is public: file Hello.java must contain class Hello. The opening { starts the class body, which contains everything that belongs to the class.

public static void main(String[] args) {

This is the famous main method — the entry point for any standalone Java program. When you run java Hello, the JVM looks for exactly this signature on the class you named. If it can't find it, you get an error like "Error: Main method not found in class Hello."

Each piece of the signature matters:

  • public — callable from outside the class. The JVM is outside the class, so it needs public to call this method.
  • static — belongs to the class itself, not to an instance. The JVM doesn't create a Hello object before calling main; it just calls the method directly on the class.
  • void — returns nothing. The method doesn't produce a value.
  • main — the magic name the JVM looks for. Spell it any other way and the JVM won't find your entry point.
  • String[] args — command-line arguments. The JVM packages anything you typed after the class name into this array.

If any of these is wrong, the JVM either won't find your main method or will fail to start. There's some flexibility: String[] args can also be written String args[] or String... args (varargs — Chapter 8). The parameter name doesn't have to be args; it just conventionally is.

System.out.println("Hello, World!");

This is the actual work — printing a line of text. Three pieces:

  • System — a built-in class in the java.lang package. It's auto-imported, so you don't need an import statement to use it.
  • out — a public static field on System representing standard output (typically your terminal). It's an instance of PrintStream.
  • println — a method on PrintStream that prints its argument and adds a newline. There's also print (no newline) and printf (formatted printing).

The semicolon at the end terminates the statement. Java requires semicolons after every statement — forgetting them is a common early mistake. Unlike Python or JavaScript-with-ASI, the compiler will not forgive you.

Variations on print

System.out.println("with newline");    // adds \n at the end
System.out.print("no newline ");        // no newline
System.out.print("continues here\n");  // manual newline

// printf — C-style formatted printing:
System.out.printf("Hi %s, you are %d years old%n", "Alice", 30);
// Hi Alice, you are 30 years old

// Common format specifiers:
//   %s   string
//   %d   integer
//   %f   floating-point  (%.2f = 2 decimal places)
//   %n   platform-specific newline (preferred over \n in printf)
//   %b   boolean

Curly braces and code structure

Java uses { } to delimit blocks. The class body is inside one pair of braces; the method body is inside another, nested inside the class. The braces are how the compiler knows where things start and end.

Indentation isn't significant the way it is in Python — the compiler ignores it. But consistent indentation (4 spaces is conventional in Java; some shops use tabs or 2 spaces) is critical for readability. Any modern IDE will format automatically.

Two equivalent styles you'll see in the wild:

// K&R style (brace on same line) — Oracle's official style, most common:
public class Hello {
    public static void main(String[] args) {
        System.out.println("Hello");
    }
}

// Allman style (brace on next line) — common in older codebases:
public class Hello
{
    public static void main(String[] args)
    {
        System.out.println("Hello");
    }
}

Pick one and stick with it. K&R is more idiomatic in modern Java; it's what your IDE will produce by default.

Reading command-line arguments

The args array holds anything you typed after the class name on the command line. Each space-separated token becomes one element.

public class Greet {
    public static void main(String[] args) {
        if (args.length > 0) {
            System.out.println("Hello, " + args[0] + "!");
        } else {
            System.out.println("Hello, stranger!");
        }
    }
}
$ java Greet Alice
Hello, Alice!

$ java Greet
Hello, stranger!

$ java Greet Alice Bob Carol
Hello, Alice!
# Only printed Alice — we only check args[0]

If you want to process all the arguments, you'd use a loop (covered properly in Chapter 7). Here's a preview:

public class GreetAll {
    public static void main(String[] args) {
        if (args.length == 0) {
            System.out.println("Hello, stranger!");
            return;
        }
        for (String name : args) {
            System.out.println("Hello, " + name + "!");
        }
    }
}
$ java GreetAll Alice Bob Carol
Hello, Alice!
Hello, Bob!
Hello, Carol!

Comments

Java has three comment forms. The first two are general-purpose; the third is special — it generates API documentation.

// Single-line comment. Goes to the end of the line.

/*
  Multi-line comment.
  Useful for blocks of explanation
  spanning several lines.
*/

/**
 * Javadoc comment — used to generate API documentation with the javadoc tool.
 * Goes immediately above classes, methods, and fields.
 *
 * @param x the first number
 * @param y the second number
 * @return the sum of x and y
 */
public int add(int x, int y) {
    return x + y;
}

Javadoc comments use special tags (@param, @return, @throws) that the javadoc tool extracts to produce HTML documentation — this is exactly how Oracle's API docs are generated. We'll see them throughout the book.

Running without compiling separately

Since Java 11, you can run a single-file source program directly with java Hello.java — no separate javac step. The JVM compiles in memory and runs immediately. Handy for quick scripts and one-off programs, but for anything multi-file you'll still want the normal compile-then-run workflow (typically managed by an IDE or build tool).

Chapter 4 — Variables and Data Types

Java is a statically-typed language: every variable has a declared type known at compile time. This catches a huge class of bugs before your code ever runs. In this chapter we'll cover Java's two big families of types (primitive and reference), the eight primitive types in detail, type conversion, the modern var keyword, and how to name variables well.

Declaring a variable

The basic pattern is type identifier = value; — type first, then the variable name, then optionally an initializer.

int age = 30;            // declare an int and initialize it
String name = "Java";    // declare a String
double price;            // declare without initializing
price = 9.99;            // assign later

You can also declare multiple variables of the same type in one statement, though this is generally discouraged for readability:

// Legal but not recommended:
int a = 1, b = 2, c = 3;

// Preferred — one per line:
int a = 1;
int b = 2;
int c = 3;
Initialize before reading

A local variable must be assigned before you read from it — the compiler enforces this. Trying to use an uninitialized local variable is a compile error: variable might not have been initialized. Fields (variables declared inside a class but outside any method) get default values automatically — we'll see this in Part III.

Two families of types

Java's type system splits cleanly into two categories:

  • Primitive types — store a value directly. Eight of them, all built into the language. Fixed size, no methods, no inheritance. Very efficient.
  • Reference types — everything else. Variables hold a reference (think pointer) to an object on the heap. Includes String, arrays, your own classes, and every class in the standard library.

This distinction shapes a lot of Java behavior — how variables are compared, what "null" means, how memory works. We'll see it come up repeatedly.

The eight primitive types

Primitives store a raw value directly, not a reference. They're efficient and ubiquitous. Each has a fixed size in bits, which determines its range.

Integer types

byte b = 127;            // 8-bit signed, range -128 to 127
short s = 32_000;        // 16-bit signed, range -32,768 to 32,767
int i = 2_000_000_000;   // 32-bit signed, range ~ ±2.1 billion
long l = 9_000_000_000L; // 64-bit signed, range ~ ±9.2 quintillion

// Note the L suffix on long literals.
// Without it, the compiler treats 9_000_000_000 as int — overflow error.

int is the default for whole numbers. You'll use it for almost everything. long when you need bigger range (timestamps, large counts, file sizes). byte and short are rare — usually only used for memory-sensitive data like binary file content or network packets.

Floating-point types

float f = 3.14f;         // 32-bit IEEE 754, ~7 decimal digits of precision
double d = 3.141592653;  // 64-bit IEEE 754, ~15 decimal digits of precision

// Note the f suffix on float literals.
// Without it, 3.14 is a double, and assigning to float fails:
//   float bad = 3.14;       // ERROR
//   float ok  = 3.14f;      // OK
//   float ok2 = (float)3.14; // OK with explicit cast

double is the default for decimal numbers. You'll use it nearly always. float is rare; it shows up in graphics code, machine learning, or anywhere memory bandwidth matters more than precision.

Boolean and character

boolean active = true;   // either true or false. No numeric value.
char letter = 'A';       // 16-bit Unicode character. SINGLE quotes!

// 'A' is a char (single quotes).
// "A" is a String (double quotes). Completely different types.

char unicode = '\u00E9';  // é — Unicode escape
char tab = '\t';
char newline = '\n';

char is 16 bits because Java predates wide adoption of Unicode beyond the Basic Multilingual Plane (the first 65,536 code points). Modern Java handles supplementary characters using surrogate pairs — two chars representing one logical character. For most everyday text this Just Works.

Default values

Local variables have no default — you must initialize them. But fields (class-level variables) get sensible defaults automatically:

// Default values for fields (not local variables):
//   byte, short, int, long → 0
//   float, double          → 0.0
//   boolean                → false
//   char                   → '\u0000' (the null character)
//   reference types        → null

Numeric literals and readability

int basic = 1000000;
int readable = 1_000_000;        // underscores allowed (Java 7+)
long big = 9_999_999_999L;       // L suffix for long
double pi = 3.14159;
float f = 1.5f;                  // f suffix for float
double exp = 1.5e10;             // scientific notation: 1.5 × 10^10

// Different number bases:
int decimal = 42;
int hex     = 0x2A;              // 0x prefix → hexadecimal (16-based)
int octal   = 052;               // leading 0 → octal (8-based)
int binary  = 0b101010;          // 0b prefix → binary (2-based)

// All four represent the value 42.

Underscores can go anywhere between digits — they're purely for readability. The compiler ignores them. Common uses:

long creditCard = 1234_5678_9012_3456L;   // group by 4
int phone = 555_867_5309;                  // group like a phone number
int hex = 0xFF_EC_DE_5E;                   // group hex by byte

Floating-point quirks

Floats and doubles use binary representation under the hood (IEEE 754). Many decimal values that look simple to us can't be represented exactly in binary, leading to tiny rounding errors:

double a = 0.1 + 0.2;
System.out.println(a);                  // 0.30000000000000004
System.out.println(a == 0.3);           // false (!!)

// The reason: 0.1 cannot be represented exactly in binary,
// any more than 1/3 can be represented exactly in decimal.
// Each value is the closest binary approximation, and the
// approximation errors don't cancel out perfectly.

There are also special floating-point values for unusual results:

double pos = 1.0 / 0.0;
System.out.println(pos);            // Infinity

double neg = -1.0 / 0.0;
System.out.println(neg);            // -Infinity

double nan = 0.0 / 0.0;
System.out.println(nan);            // NaN (Not a Number)

// NaN is special — it's not equal to itself!
System.out.println(nan == nan);     // false
System.out.println(Double.isNaN(nan)); // true — the correct way to check
Never use double for money

Use BigDecimal (from the java.math package) for currency, accounting, or anywhere rounding errors are unacceptable. BigDecimal stores exact decimal values and lets you control rounding mode. It's slower than double, but correctness matters more than speed in financial code.

Reference types

Anything that isn't a primitive is a reference type — its variable holds a reference (think pointer) to an object stored on the heap. String is the most common reference type and gets some special syntactic sugar:

String greeting = "Hello";
String empty = "";
String maybe = null;                  // null = 'no object'
int length = greeting.length();       // call a method — primitives can't do this

A few important consequences of being a reference type:

  • Reference types can be null. Primitives can never be null.
  • Reference types have methods. "hello".toUpperCase() works because String is a reference type.
  • Comparison with == compares references, not contents. For value equality, use .equals() — covered in Chapter 16.
  • Reference types live on the heap and are garbage collected. The variable on the stack just holds the reference.

var — local variable type inference (Java 10+)

Java 10 introduced var for local variables. The compiler infers the type from the initializer, so you don't have to repeat yourself:

var count = 10;              // inferred as int
var name = "Alice";          // inferred as String
var price = 9.99;            // inferred as double
var list = new ArrayList<String>();  // inferred as ArrayList<String>

// var only works for local variables, only with an initializer:
// var x;             ← ERROR: needs initializer
// var n = null;      ← ERROR: can't infer from null alone
// var lambda = () -> 42;  ← ERROR: can't infer lambda's functional interface

var is not dynamic typing. The variable's type is still fixed at compile time; you just didn't write it out. Once inferred, the type can't change:

var x = 10;       // x is int
x = "hello";      // ERROR — x is int, not String

// Equivalent to:
int x = 10;
x = "hello";      // same error

When to use var:

  • When the type is obvious from the right side: var list = new ArrayList<String>();
  • When the type would be very long: var iter = map.entrySet().iterator();
  • When the type adds no information: var name = user.getName();

When NOT to use var:

  • When the right side is opaque: var x = process(); — now the reader has to look up what process() returns.
  • When you want to upcast: List<String> list = new ArrayList<>(); explicitly says "I'll treat this as a List"; var would make it specifically an ArrayList.
  • For primitives where the literal is ambiguous: var x = 0; is int, but you might have wanted long.

Type conversion

Going between numeric types is common. Java distinguishes between safe automatic conversions (widening) and potentially unsafe ones (narrowing, which require explicit casts).

Widening — automatic

// Going from a smaller type to a larger one — automatic:
int i = 100;
long l = i;          // int → long ✓
double d = i;        // int → double ✓
float f = l;         // long → float ✓ (may lose precision but no overflow)

// The chain: byte → short → int → long → float → double
// Each step up is safe.

Narrowing — explicit cast required

// Going from a larger type to a smaller one — explicit, may lose data:
double d = 9.99;
int i = (int) d;     // i is 9 (decimal part truncated, not rounded!)

// Narrowing can overflow silently:
long big = 200L;
byte b = (byte) big; // 200 is outside byte range (-128 to 127)
// b is -56 — the bits got truncated to fit in 8 bits, interpreted as signed
Truncation vs rounding

Casting from double to int truncates toward zero — it doesn't round. (int) 9.99 is 9. (int) -9.99 is -9 (not -10). For rounding, use Math.round(9.99), which returns 10.

Conversion in expressions

When you combine values of different types in one expression, Java promotes the smaller to the larger:

int i = 5;
double d = 2.0;
double result = i + d;       // i is widened to double, result is 7.0

// Watch out — integer division before promotion:
double bad = 5 / 2;          // 2.0 (5/2 is integer division → 2, then widened)
double good = 5 / 2.0;       // 2.5 (at least one operand is double from the start)
double also = (double) 5 / 2;  // 2.5 — cast first, then divide

Constants with final

final double PI = 3.14159;
final int MAX_USERS = 100;

// PI = 3.14;       ← ERROR: cannot assign a value to final variable PI

// 'final' means assigned exactly once.
// You can defer the assignment, but then you can only do it once:
final int x;
if (condition) {
    x = 1;
} else {
    x = 2;
}
// x is now final and cannot be reassigned.

The convention is to name constants in UPPER_SNAKE_CASE. final on a primitive means the value can't change. final on a reference means the reference can't be reassigned, but the object it points to can still be modified — important distinction we'll revisit:

final List<String> names = new ArrayList<>();
names.add("Alice");   // OK — mutating the list, not reassigning the variable
// names = new ArrayList<>();  // ERROR — cannot reassign the reference

Naming conventions

Java has strong conventions. Following them helps everyone read your code:

  • Variables and methods: camelCase — userName, calculateTotal, isActive.
  • Classes and interfaces: PascalCase — UserAccount, OrderService, Comparable.
  • Constants: UPPER_SNAKE_CASE — MAX_RETRY_COUNT, DEFAULT_TIMEOUT.
  • Packages: all lowercase — com.example.app.
  • Generic type parameters: single uppercase letter — T, E, K, V, R.

The rules the compiler enforces:

  • Names must start with a letter, $, or _ — not a digit.
  • Subsequent characters can be letters, digits, $, or _.
  • Case sensitive — count and Count are different variables.
  • Cannot be a reserved word: class, int, public, return, and about 50 others.
  • No length limit, but keep them readable.

Good names communicate intent:

// Bad:
int x;
int n;
List<String> list;
boolean flag;

// Good:
int retryCount;
int customerAge;
List<String> customerNames;
boolean isActive;

// Use full words. Modern IDEs auto-complete — 'usrCnt' isn't faster to type than
// 'userCount', and it's harder to read.

Chapter 5 — Operators and Expressions

Operators are how you combine and transform values. Java has the usual suspects from C-family languages plus a few subtleties worth knowing — especially around integer division, string concatenation, short-circuit evaluation, and operator precedence. By the end of this chapter you'll be able to read complex Java expressions confidently.

Arithmetic operators

int a = 10, b = 3;
System.out.println(a + b);   // 13   addition
System.out.println(a - b);   // 7    subtraction
System.out.println(a * b);   // 30   multiplication
System.out.println(a / b);   // 3    integer division (truncates toward zero)
System.out.println(a % b);   // 1    remainder (modulo)

double x = 10, y = 3;
System.out.println(x / y);   // 3.3333... floating-point division

Most of these are unsurprising. The two that catch beginners are / with integers and %.

Integer division gotcha

When both operands are integer types, / performs integer division — it truncates toward zero. The result is always an integer:

10 / 3      // 3 (not 3.33)
9 / 3       // 3 (exact)
8 / 3       // 2 (truncated)
-10 / 3     // -3 (truncated toward zero, not -4)

// To get a floating-point result, make at least one operand a double:
10.0 / 3        // 3.3333...
10 / 3.0        // 3.3333...
(double) 10 / 3 // 3.3333...
When this matters most

Averaging integers is a classic place this bites people: (a + b) / 2 gives integer division. For correct averaging of integers as doubles, divide by 2.0 — and watch out for overflow if a + b might exceed int range, in which case use long arithmetic.

The modulo operator

10 % 3      // 1   (10 = 3*3 + 1, remainder is 1)
9 % 3       // 0   (divides evenly)
15 % 4      // 3
-7 % 3      // -1  (the sign matches the dividend in Java)

// Common uses:
n % 2 == 0          // is n even?
n % 10              // last digit of n
n / 10              // drop the last digit
year % 4 == 0       // potential leap year (simplified)

Modulo also works on doubles, where it gives the fractional remainder:

10.5 % 3.0      // 1.5
7.25 % 2.0      // 1.25

Increment and decrement

int i = 5;
i++;            // post-increment: i becomes 6
++i;            // pre-increment:  i becomes 7
i--;            // post-decrement: i becomes 6
--i;            // pre-decrement:  i becomes 5

When used as a standalone statement, post and pre forms behave identically — both just change the variable. The difference appears when they're used as part of a larger expression: the post form returns the value before the change, and the pre form returns the value after the change.

int x = 5;
int y = x++;    // y gets x's current value (5), THEN x becomes 6
// After: x = 6, y = 5

int a = 5;
int b = ++a;    // a becomes 6 first, THEN b gets the new value (6)
// After: a = 6, b = 6

A trickier example:

int n = 5;
int result = n++ + ++n;
// Step by step:
//   n++  →  evaluates to 5, then n becomes 6
//   ++n  →  n becomes 7, then evaluates to 7
//   result = 5 + 7 = 12
// Final: n = 7, result = 12
Don't mix increments in complex expressions

Code like n++ + ++n is technically well-defined in Java (unlike in C/C++ where it's undefined behavior), but it's genuinely hard to read. Put the increments on their own lines. Your future self will thank you.

Compound assignment operators

int n = 10;
n += 5;     // n = n + 5  →  15
n -= 3;     // n = n - 3  →  12
n *= 2;     // n = n * 2  →  24
n /= 4;     // n = n / 4  →  6
n %= 4;     // n = n % 4  →  2

// String concatenation:
String s = "Hello";
s += ", World!";   // s = "Hello, World!"

// Bitwise compound assignments too:
int flags = 0;
flags |= 0b100;    // set bit 2

Compound operators are slightly more than syntactic sugar — they include an implicit cast. byte b = 10; b += 5; compiles fine, but byte b = 10; b = b + 5; does NOT — the right side is an int, and assigning int to byte requires an explicit cast. The compound form casts for you.

Comparison operators

Comparison operators return boolean:

int a = 5, b = 10;
a == b      // false   equality
a != b      // true    inequality
a < b       // true    less than
a > b       // false   greater than
a <= b      // true    less than or equal
a >= b      // false   greater than or equal

The == trap with objects

For primitives, == compares values — what you'd expect. For objects, it compares references — whether two variables point to the same object in memory, not whether they're 'equal' in any meaningful sense.

String a = new String("hello");
String b = new String("hello");
System.out.println(a == b);          // false — different objects
System.out.println(a.equals(b));     // true — same content

// String literals are special — they're pooled, so identical literals are
// the same object:
String x = "hello";
String y = "hello";
System.out.println(x == y);          // true — same pooled object
// But you can't rely on this; always use .equals() for Strings.
The rule for objects

Always use .equals() for object comparison. == is correct only when you specifically mean "are these the exact same object?" — which is rare. We'll cover this in detail for Strings (Chapter 16) and writing your own equals (Chapter 11).

Logical operators

boolean a = true, b = false;
a && b      // false   AND (short-circuit)
a || b      // true    OR  (short-circuit)
!a          // false   NOT
a ^ b       // true    XOR (exclusive or — true if exactly one is true)

Short-circuit evaluation

&& and || are short-circuit: they only evaluate the right operand if necessary. This is more than a performance optimization — it's frequently used for safety:

// Safe null check — if user is null, isActive() is never called:
if (user != null && user.isActive()) {
    grantAccess(user);
}

// Without short-circuit, user.isActive() would throw NullPointerException
// when user is null.

// || version:
if (config == null || config.isDefault()) {
    useDefaults();
}
// If config is null, we use defaults without trying to call isDefault().

Java also has non-short-circuit versions: & and |. They always evaluate both sides. You'll rarely use them as logical operators (they're more commonly bitwise — see below).

Bitwise operators

These operate on the individual bits of integer types. Rare in everyday application code, but useful for flags, low-level work, encoding/decoding, and interview questions.

int a = 0b1100;   // 12 in decimal
int b = 0b1010;   // 10

a & b    // 0b1000 = 8     bitwise AND (both bits set)
a | b    // 0b1110 = 14    bitwise OR  (either bit set)
a ^ b    // 0b0110 = 6     bitwise XOR (bits differ)
~a       // ...11110011    bitwise NOT (flips all bits — gives -13 here)

a << 1   // 0b11000 = 24   left shift (effectively multiply by 2)
a << 3   // 0b1100000 = 96 left shift by 3 (multiply by 8)
a >> 1   // 0b110 = 6      arithmetic right shift (divide by 2, preserves sign)
a >>> 1  //                logical right shift (fills with 0)

A practical example — flag sets

// Define flags as powers of two so each occupies a distinct bit:
static final int READ    = 1 << 0;   // 0b001 = 1
static final int WRITE   = 1 << 1;   // 0b010 = 2
static final int EXECUTE = 1 << 2;   // 0b100 = 4

// Combine flags with OR:
int perms = READ | WRITE;            // 0b011 = 3

// Check a flag with AND:
boolean canRead = (perms & READ) != 0;     // true
boolean canExec = (perms & EXECUTE) != 0;  // false

// Add a flag with OR:
perms |= EXECUTE;                    // perms is now 0b111 = 7

// Remove a flag with AND + NOT:
perms &= ~WRITE;                     // clear the WRITE bit

// Toggle a flag with XOR:
perms ^= READ;                       // flip the READ bit

This is the classic technique used by file permissions on Unix and by many low-level APIs. In modern Java application code you'd usually use an EnumSet instead, but the bitwise approach is faster and uses less memory when you really need it.

The ternary operator

int age = 20;
String status = (age >= 18) ? "adult" : "minor";
// status = "adult"

// Form: condition ? valueIfTrue : valueIfFalse
// Equivalent to:
String status2;
if (age >= 18) {
    status2 = "adult";
} else {
    status2 = "minor";
}

The ternary is great for short, clear choices. It gets hard to read when nested or when the branches are long — use a full if in those cases.

// Clear:
int max = (a > b) ? a : b;
String label = (count == 1) ? "item" : "items";

// Hard to read — use if/else:
String result = (x > 0) ? (y > 0 ? "Q1" : "Q4") : (y > 0 ? "Q2" : "Q3");

String concatenation with +

The + operator is overloaded for Strings — it builds a new String by joining its operands. Non-String operands get converted via toString().

String name = "Alice";
int age = 30;
String message = "Hello, " + name + "! You are " + age + ".";
// "Hello, Alice! You are 30."

// Watch out for evaluation order. + is left-associative:
System.out.println(1 + 2 + " = sum");    // "3 = sum"
// Steps: 1 + 2 = 3 (int), then 3 + " = sum" = "3 = sum" (String)

System.out.println("sum = " + 1 + 2);    // "sum = 12"  (not "sum = 3"!)
// Steps: "sum = " + 1 = "sum = 1" (String), then + 2 = "sum = 12" (String)

// Force int math first with parentheses:
System.out.println("sum = " + (1 + 2));   // "sum = 3"

Operator precedence

When an expression mixes operators, precedence determines what happens first. Java's precedence rules largely follow what you'd expect from math: multiplication before addition, comparisons before logical operators.

From highest to lowest precedence (operators in the same row are equal):

  • ++ -- (postfix)
  • ++ -- + - ! ~ (unary)
  • (cast) (type) new
  • * / %
  • + -
  • << >> >>>
  • < <= > >= instanceof
  • == !=
  • &
  • ^
  • |
  • &&
  • ||
  • ?:
  • = += -= *= /= ...

Some examples worked through:

1 + 2 * 3       // 7  (multiplication before addition)
(1 + 2) * 3     // 9  (parentheses override)

5 + 3 == 8      // true  (+ binds tighter than ==)
5 < 3 || true   // true  (< before ||)

!true && false  // false (! before &&)
!(true && false) // true  (parens force AND first, then NOT)
Use parentheses liberally

Even when precedence is on your side, parentheses make intent obvious to readers. (a + b) * c beats a + b * c when you mean the former — you don't want readers reaching for the precedence table. Keep them.

Associativity

Most operators are left-associative: a - b - c is (a - b) - c. Assignment is right-associative: a = b = c is a = (b = c).

int a, b, c;
a = b = c = 5;     // c = 5; b = c (5); a = b (5). All three are 5.

// Left-associative subtraction:
10 - 3 - 2     // (10 - 3) - 2 = 5
10 - (3 - 2)   // 10 - 1 = 9

Control Flow & Methods

Chapter 6 — Conditional Statements

Programs need to make decisions. This chapter covers Java's four ways of branching: the if/else family, the classic switch statement, modern switch expressions (Java 14+), and pattern matching for switch (Java 21+). By the end you'll know not just the syntax but which form to reach for in different situations.

The if statement

The most basic conditional. An if runs a block of code only when a boolean condition is true:

int temperature = 75;

if (temperature > 80) {
    System.out.println("It's hot.");
}

If the condition is false, the block is skipped entirely. The condition must be a boolean — Java does not treat zero, null, or empty strings as "falsy" the way some languages do.

int n = 0;
if (n) { ... }              // COMPILE ERROR — n is int, not boolean
if (n != 0) { ... }         // OK — explicit comparison gives a boolean

String s = null;
if (s) { ... }              // COMPILE ERROR
if (s != null) { ... }      // OK

if-else

int score = 75;

if (score >= 50) {
    System.out.println("Pass");
} else {
    System.out.println("Fail");
}

Exactly one of the two branches runs.

if-else if-else

int score = 75;

if (score >= 90) {
    System.out.println("Excellent");
} else if (score >= 70) {
    System.out.println("Good");
} else if (score >= 50) {
    System.out.println("Pass");
} else {
    System.out.println("Fail");
}

The conditions are checked top to bottom. As soon as one is true, that block runs and the rest are skipped. The final else is optional and acts as a catch-all.

Notice the order matters. If you reversed it (checking >= 50 first), everyone would get "Pass" because that condition is true for scores 50+ — including 90+.

Braces — optional but always include them

// Legal Java — no braces, single statement only:
if (x > 0)
    System.out.println("positive");

// But this is a bug waiting to happen:
if (x > 0)
    System.out.println("positive");
    x = 0;                          // Always runs! Not part of the if!

// Always use braces:
if (x > 0) {
    System.out.println("positive");
    x = 0;                          // clearly part of the if
}
Style: always braces

Most Java style guides require braces on every if/else, even for single statements. The cost is a few characters; the benefit is eliminating an entire category of bugs when someone (often you) later adds a second line and forgets to add braces. The Apple goto-fail bug is a famous example of what can go wrong.

Nested if statements

if (user != null) {
    if (user.isActive()) {
        if (user.hasPermission("read")) {
            displayContent();
        }
    }
}

Nesting works, but it gets ugly fast. Two common ways to flatten this kind of code:

1. Combine with logical operators:

if (user != null && user.isActive() && user.hasPermission("read")) {
    displayContent();
}
// Short-circuit evaluation means we won't call isActive() if user is null.

2. Use guard clauses (early returns):

if (user == null) return;
if (!user.isActive()) return;
if (!user.hasPermission("read")) return;

displayContent();
// The main flow is at the bottom, unindented. Special cases handled upfront.

Guard clauses are especially popular in modern Java because they keep the indentation level low and put the 'happy path' at the bottom, where the eye naturally lands.

The classic switch statement

When you have many branches based on a single value, an if/else-if chain gets repetitive. The switch statement handles this more cleanly:

int day = 3;
switch (day) {
    case 1:
        System.out.println("Monday");
        break;
    case 2:
        System.out.println("Tuesday");
        break;
    case 3:
        System.out.println("Wednesday");
        break;
    case 4:
        System.out.println("Thursday");
        break;
    case 5:
        System.out.println("Friday");
        break;
    default:
        System.out.println("Weekend");
}

The expression in parentheses is evaluated once, then control jumps to the matching case label. default handles anything not explicitly listed. It's optional but usually a good idea.

The fall-through trap

Without break, execution falls through to the next case. This is a frequent source of bugs in the classic switch syntax. Modern switch expressions (next section) fix this — but in classic switch, you must remember the break:

int day = 3;
switch (day) {
    case 1:
        System.out.println("Monday");
        // No break! Falls through to case 2 also.
    case 2:
        System.out.println("Tuesday");
        break;
}
// If day is 1, prints both "Monday" AND "Tuesday". Probably not intended.

Intentional fall-through is occasionally useful — for example, when multiple cases should do the same thing:

switch (day) {
    case 1: case 2: case 3: case 4: case 5:
        System.out.println("Weekday");
        break;
    case 6: case 7:
        System.out.println("Weekend");
        break;
}
// All weekdays fall through to the same body.

What can you switch on?

  • Integer types: byte, short, int, char
  • Their wrapper classes: Byte, Short, Integer, Character
  • String (since Java 7)
  • Enums
  • Patterns of these (since Java 21) — covered shortly

You cannot switch on long, float, double, or boolean.

Switching on strings

String command = "start";
switch (command) {
    case "start":
        System.out.println("Starting");
        break;
    case "stop":
        System.out.println("Stopping");
        break;
    case "restart":
        System.out.println("Restarting");
        break;
    default:
        System.out.println("Unknown: " + command);
}

String switches are case-sensitive and use .equals() internally. Be aware: switching on null throws NullPointerException in the classic switch (modern switch expressions have a fix).

Switching on enums

enum Color { RED, GREEN, BLUE }

Color c = Color.GREEN;
switch (c) {
    case RED:    System.out.println("Stop"); break;
    case GREEN:  System.out.println("Go");   break;
    case BLUE:   System.out.println("Cool"); break;
}
// Note: no need to write 'Color.RED' inside cases — the type is known.

Switch expressions (Java 14+)

Modern Java has a new switch syntax that's concise, returns a value, and never falls through. Use this style going forward — it's clearer and safer.

Switch expression — arrow form:

int day = 3;
String name = switch (day) {
    case 1 -> "Monday";
    case 2 -> "Tuesday";
    case 3 -> "Wednesday";
    case 4 -> "Thursday";
    case 5 -> "Friday";
    case 6, 7 -> "Weekend";       // multiple labels per case
    default -> "Invalid";
};
System.out.println(name);   // Wednesday

Three big differences from the classic switch: it returns a value (note the semicolon at the end of the whole expression), it uses arrows instead of colons, and there's no fall-through to worry about. Each case is independent.

Multi-statement bodies with yield

If a case body needs multiple statements, use a block and the new yield keyword to return a value:

String describe = switch (day) {
    case 1, 2, 3, 4, 5 -> "Weekday";
    case 6, 7 -> {
        System.out.println("Logging weekend access");
        yield "Weekend";
    }
    default -> "Invalid";
};

Exhaustiveness

A switch expression must cover all possible input values — the compiler enforces this. For an enum, that means handling every constant. For other types, you'll need a default:

enum Color { RED, GREEN, BLUE }

String name = switch (color) {
    case RED   -> "red";
    case GREEN -> "green";
    case BLUE  -> "blue";
    // no default needed — all cases covered
};

// If we add YELLOW to the enum and forget to update this switch,
// the compiler will tell us! Wonderful for refactoring safety.
Classic switch vs switch expression

Use the switch expression (arrow form) when you want to return a value or want exhaustiveness checking. Use the classic switch statement only when doing side effects with no return value, or when you genuinely want fall-through. For most new code, switch expressions are the better default.

Pattern matching for switch (Java 21+)

Java 21 made pattern matching for switch a stable feature. You can switch on the type of an object and bind a variable in one step. Combined with sealed types (Chapter 30), this is genuinely powerful.

static String describe(Object obj) {
    return switch (obj) {
        case Integer i  -> "int: " + i;
        case String s   -> "string of length " + s.length();
        case int[] arr  -> "int array of length " + arr.length;
        case null       -> "null value";
        default         -> "unknown";
    };
}

// Each case binds a new variable (i, s, arr) typed to that case.
// No explicit casts needed.

Guarded patterns with when

String result = switch (obj) {
    case Integer i when i < 0  -> "negative int: " + i;
    case Integer i when i == 0 -> "zero";
    case Integer i             -> "positive int: " + i;
    case String s when s.isEmpty() -> "empty string";
    case String s                  -> "string: " + s;
    default -> "other";
};

// The 'when' clause adds an extra condition.
// Note ordering matters — more specific cases must come first.

Handling null in switch expressions

Pre-Java 21 switch threw NullPointerException on null. Modern switch expressions can handle null explicitly:

String greet(String name) {
    return switch (name) {
        case null    -> "Hello, stranger";
        case ""      -> "Hello, anonymous";
        case "admin" -> "Hello, administrator";
        default      -> "Hello, " + name;
    };
}

A switch statement performs side effects but doesn't yield a value. It uses colons and requires break to prevent fall-through. Use it when each case does some action (print, update state) with no value to return.

Chapter 7 — Loops

Loops let you repeat work. Java has four: while, do-while, the classic for, and the enhanced for-each. This chapter covers all four, when to use each, and how to control loops with break, continue, and labels. You'll use enhanced for and the classic for most often.

The while loop

The simplest loop. It checks a condition; if true, runs the body; then checks again.

int n = 1;
while (n <= 5) {
    System.out.println(n);
    n++;
}
// Prints 1, 2, 3, 4, 5
// Then n is 6, the condition fails, and the loop exits.

Use while when you don't know in advance how many iterations you need — the loop continues as long as some condition holds.

A more realistic example — reading until a sentinel value:

Scanner sc = new Scanner(System.in);
int total = 0;
int n = sc.nextInt();
while (n != 0) {              // 0 marks the end of input
    total += n;
    n = sc.nextInt();
}
System.out.println("Total: " + total);

Infinite loops and intentional ones

// Accidentally infinite — a classic bug:
int i = 0;
while (i < 10) {
    System.out.println(i);
    // Forgot i++!  Loop runs forever.
}

// Intentionally infinite — for event loops, servers, etc.:
while (true) {
    Event event = waitForEvent();
    if (event.isShutdown()) break;
    handle(event);
}

The do-while loop

Like while, but checks the condition after the body. The body always runs at least once.

int input;
do {
    System.out.print("Enter a positive number: ");
    input = sc.nextInt();
} while (input <= 0);

// The prompt always shows at least once.
// We keep asking until they enter something positive.
// Note the semicolon after while — required here, unlike while loops.

do-while is less common than plain while. The most common use is interactive prompts where you need to read input before you can decide whether to stop.

The classic for loop

The most flexible loop construct. Three semicolon-separated parts in the header: initialization, condition, and update.

for (int i = 0; i < 10; i++) {
    System.out.println(i);
}
// Prints 0 through 9.

// Anatomy:
//   int i = 0;     ← initialization, runs once before the loop starts
//   i < 10;        ← condition, checked before each iteration
//   i++            ← update, runs after each iteration

Variations on the basic pattern:

// Count up by 2s:
for (int i = 0; i < 20; i += 2) {
    System.out.println(i);   // 0, 2, 4, ..., 18
}

// Count down:
for (int i = 10; i > 0; i--) {
    System.out.println(i);   // 10, 9, 8, ..., 1
}

// Multiple variables — comma separated:
for (int i = 0, j = 10; i < j; i++, j--) {
    System.out.println(i + ", " + j);
}

// All three parts are optional — infinite loop with for:
for (;;) {
    // do something forever, until you break
}

Iterating over a range

int[] nums = {10, 20, 30, 40, 50};

// Classic for — when you need the index:
for (int i = 0; i < nums.length; i++) {
    System.out.println("index " + i + ": " + nums[i]);
}

Note nums.length — a field (no parentheses), not a method. Arrays have a length field; collections like ArrayList have a size() method. Easy mix-up.

The enhanced for (for-each) loop

The cleanest way to iterate over arrays and collections when you don't need the index. Often called the 'for-each' loop.

int[] numbers = {10, 20, 30, 40};
for (int n : numbers) {
    System.out.println(n);
}

// Read out loud: "for each int n in numbers"

// Works for any Iterable, including List, Set, etc.:
List<String> names = List.of("Alice", "Bob", "Carol");
for (String name : names) {
    System.out.println("Hello, " + name);
}

// Even works on String (with toCharArray) and Map (via entrySet/keySet/values):
for (char c : "hello".toCharArray()) {
    System.out.println(c);
}

Map<String, Integer> scores = Map.of("alice", 90, "bob", 85);
for (Map.Entry<String, Integer> e : scores.entrySet()) {
    System.out.println(e.getKey() + ": " + e.getValue());
}

When to use each loop type

  • for-each — when you just need each element. Cleanest, hardest to get wrong. First choice.
  • Classic for — when you need the index, want to iterate in reverse, want to skip elements, or are iterating over multiple things in parallel.
  • while — when iterations depend on a condition unrelated to counting (e.g., reading until end of file).
  • do-while — when you must execute the body at least once before checking.

break and continue

Two ways to alter normal loop flow. break exits the loop entirely. continue skips the rest of the current iteration and goes back to the condition check.

for (int i = 1; i <= 10; i++) {
    if (i == 5) break;        // exit the loop entirely
    if (i % 2 == 0) continue; // skip even numbers — go to next iteration
    System.out.println(i);
}
// Prints: 1, 3

Practical example — finding the first matching element:

int[] nums = {3, 7, 12, 18, 25};
int firstEven = -1;
for (int n : nums) {
    if (n % 2 == 0) {
        firstEven = n;
        break;          // found it — stop looking
    }
}
System.out.println(firstEven);   // 12

Labeled breaks for nested loops

break normally exits the innermost enclosing loop. To break out of multiple loops at once, use a label:

outer:
for (int i = 0; i < 5; i++) {
    for (int j = 0; j < 5; j++) {
        if (i * j > 6) {
            break outer;        // breaks out of BOTH loops
        }
        System.out.println(i + "," + j);
    }
}
System.out.println("done");

// continue also works with labels:
// 'continue outer' would skip to the next iteration of the outer loop.
Labels are rare

Labeled breaks are powerful but uncommon. Often, refactoring to extract a method (with an early return instead of a labeled break) gives cleaner code. Save labels for the cases where they really do read better.

Common loop patterns

Sum or accumulate

int[] nums = {1, 2, 3, 4, 5};
int sum = 0;
for (int n : nums) {
    sum += n;
}
// sum = 15

Find max or min

int[] nums = {3, 1, 4, 1, 5, 9, 2, 6};
int max = nums[0];               // start with first element as baseline
for (int i = 1; i < nums.length; i++) {
    if (nums[i] > max) {
        max = nums[i];
    }
}
// max = 9

Count matches

String s = "Hello, World!";
int vowelCount = 0;
for (char c : s.toLowerCase().toCharArray()) {
    if ("aeiou".indexOf(c) >= 0) {
        vowelCount++;
    }
}
// vowelCount = 3 (e, o, o)

Build a string

String[] words = {"hello", "world", "!"};
StringBuilder sb = new StringBuilder();
for (int i = 0; i < words.length; i++) {
    if (i > 0) sb.append(" ");
    sb.append(words[i]);
}
String result = sb.toString();   // "hello world !"

Two pointers

// Check if a string is a palindrome:
static boolean isPalindrome(String s) {
    int left = 0, right = s.length() - 1;
    while (left < right) {
        if (s.charAt(left) != s.charAt(right)) return false;
        left++;
        right--;
    }
    return true;
}

isPalindrome("racecar");   // true
isPalindrome("hello");     // false

Nested loops

// Print a multiplication table:
for (int i = 1; i <= 5; i++) {
    for (int j = 1; j <= 5; j++) {
        System.out.printf("%3d ", i * j);

Chapter 8 — Methods

Methods are how you package reusable behavior. This chapter covers method syntax, parameters, return values, overloading, varargs, scope, recursion, and one of Java's most-misunderstood concepts: pass-by-value semantics and how they interact with references.

Anatomy of a method

public static int add(int a, int b) {
    return a + b;
}

// Breaking it down:
//   public         - access modifier (visible from anywhere)
//   static         - belongs to the class, not an instance
//   int            - return type
//   add            - method name (camelCase, verb-ish)
//   (int a, int b) - parameter list
//   { ... }        - method body
//   return         - sends a value back to the caller

We'll cover public and static thoroughly in Part III. For now, just know that static lets you call a method without first creating an instance — convenient for utility functions and for everything in your main class.

Calling methods

int sum = add(3, 4);                      // sum = 7
System.out.println(add(10, 20));          // prints 30
int doubled = add(add(1, 2), add(3, 4));  // doubled = 10

// The arguments are evaluated left-to-right before the call.
// In add(add(1, 2), add(3, 4)):
//   1. add(1, 2) is called -> 3
//   2. add(3, 4) is called -> 7
//   3. add(3, 7) is called -> 10

Return values

A method's return type determines what kind of value it sends back:

// Returns an int - must end with a return statement returning an int:
public static int square(int n) {
    return n * n;
}

// Returns nothing (void) - return is optional:
public static void greet(String name) {
    System.out.println("Hello, " + name);
    // No return needed.
}

// You can use 'return' alone in a void method to exit early:
public static void describe(int n) {
    if (n < 0) {
        System.out.println("negative");
        return;                   // early exit
    }
    System.out.println("non-negative: " + n);
}

Every code path must return

// Compile error - not every path returns:
public static int classify(int n) {
    if (n > 0) {
        return 1;
    } else if (n < 0) {
        return -1;
    }
    // What about n == 0? Compiler complains.
}

// Fix - make every path return:
public static int classify(int n) {
    if (n > 0) return 1;
    if (n < 0) return -1;
    return 0;
}

Parameters

Parameters are local variables. They're initialized to the argument values when the method is called, and they go away when the method returns.

public static void doStuff(int x, int y) {
    // x and y are local - only visible inside this method
    int sum = x + y;
    // sum is also local
}

doStuff(5, 10);
// At this point x, y, sum are gone.

Method overloading

Multiple methods can share a name as long as their parameter lists differ — different types or different number of parameters. This is called overloading.

public static int add(int a, int b) {
    return a + b;
}

public static double add(double a, double b) {
    return a + b;
}

public static int add(int a, int b, int c) {
    return a + b + c;
}

public static String add(String a, String b) {
    return a + b;
}

// All four coexist. The compiler picks the right one based on the arguments:
add(1, 2);              // calls the int version -> 3
add(1.5, 2.5);          // calls the double version -> 4.0
add(1, 2, 3);           // calls the three-arg int version -> 6
add("Hello, ", "World");  // calls the String version -> "Hello, World"
Overloading vs overriding

Overloading happens at compile time - multiple methods with the same name but different parameter lists. Overriding happens at runtime - a subclass provides its own version of a method inherited from a superclass. They sound similar but are completely different mechanisms. Overriding is Chapter 11.

What makes overloads distinguishable

  • Different number of parameters: add(int) vs add(int, int)
  • Different parameter types: add(int, int) vs add(double, double)
  • Different parameter order: add(int, double) vs add(double, int)

NOT distinguishable by return type alone, or by parameter names:

// These two cannot coexist - same parameter types, different return type:
public static int compute(int n) { ... }
public static double compute(int n) { ... }  // COMPILE ERROR

// These two cannot coexist - only parameter names differ:
public static int compute(int x) { ... }
public static int compute(int y) { ... }     // COMPILE ERROR

Varargs - variable-length arguments

Sometimes you want a method that can take any number of arguments. The ... syntax - called varargs - handles this:

public static int sum(int... numbers) {
    int total = 0;
    for (int n : numbers) total += n;
    return total;
}

// Call with any number of arguments - including zero:
sum();                  // 0
sum(5);                 // 5
sum(1, 2, 3, 4, 5);     // 15

// Inside the method, 'numbers' is treated as an int[]:
//   numbers.length, numbers[0], etc.

Varargs can also accept an actual array, which is occasionally useful:

int[] arr = {1, 2, 3};
sum(arr);     // works - passes the array directly

// Or use a mix:
// sum(arr, 4);  // ERROR - once you pass an array, you can't mix

Rules for varargs:

  • Only one varargs parameter per method, and it must be the last parameter.
  • String first, int... rest is fine.
  • int... first, String last is illegal.
  • Inside the method, the varargs parameter is just an array.

Pass-by-value

This is one of the most-misunderstood aspects of Java. The rule is simple: Java is always pass-by-value. The method gets a copy of the argument. Reassigning the parameter inside the method doesn't affect the caller.

public static void tryToChange(int x) {
    x = 99;
    System.out.println("inside: x = " + x);
}

int n = 5;
tryToChange(n);
System.out.println("outside: n = " + n);

// Output:
//   inside: x = 99
//   outside: n = 5

x is a local copy of n's value. Reassigning x doesn't touch n.

Pass-by-value with references

Here's where it gets confusing. For objects, the value being copied is a reference. The method gets its own copy of the reference, but both copies point to the same underlying object. Mutating the object through that reference is visible to the caller. Reassigning the parameter is not.

public static void addItem(List<String> list) {
    list.add("new");                  // mutates the shared object - visible
    list = new ArrayList<>();          // reassigns local - invisible to caller
    list.add("won't be seen");
}

List<String> items = new ArrayList<>();
items.add("first");
addItem(items);
System.out.println(items);    // [first, new]

// 'new' was added because the method mutated the shared list.
// 'won't be seen' was added to a different list that was discarded
// when the method returned.

Visualizing what happens:

Before call:
  items --> [first]            (the list lives on the heap)

During call, list adds "new":
  items --> [first, new]       (same object, modified)
  list  --^

After 'list = new ArrayList<>()':
  items --> [first, new]       (unchanged from caller's view)
  list  --> [] (new object)

After return:
  items --> [first, new]
  (list is gone, the second ArrayList is garbage)
The right mental model

Java passes references by value. The variable in the caller and the parameter in the method are two different variables that initially point to the same object. What you do to the object is shared; what you do to the parameter variable itself is not.

Variable scope

Scope is the region of code where a variable is accessible. The rule in Java: a variable is accessible from its declaration until the end of the enclosing block (a pair of curly braces).

public static void example() {
    int x = 10;            // scope: the whole method
    if (x > 0) {
        int y = 20;        // scope: the if block only
        System.out.println(x + y);   // both x and y visible here
    }
    // y is NOT accessible here - out of scope.
    System.out.println(x);  // x still visible - same scope as declaration
}

Loop variables in a classic for are scoped to the loop:

for (int i = 0; i < 10; i++) {
    // i is visible here
}
// i is NOT visible here.

// If you need the loop variable after the loop, declare it outside:
int i;
for (i = 0; i < 10; i++) {
    if (someCondition(i)) break;
}
System.out.println("stopped at " + i);

Recursion

A method that calls itself is recursive. Every recursion needs a base case that doesn't recurse - otherwise it runs forever (or until the JVM's stack overflows).

public static int factorial(int n) {
    if (n <= 1) return 1;              // base case
    return n * factorial(n - 1);       // recursive step
}

factorial(5);
// = 5 * factorial(4)
// = 5 * 4 * factorial(3)
// = 5 * 4 * 3 * factorial(2)
// = 5 * 4 * 3 * 2 * factorial(1)
// = 5 * 4 * 3 * 2 * 1
// = 120

Recursion vs iteration

// Same function, two ways.

// Recursive:
static int factR(int n) {
    if (n <= 1) return 1;
    return n * factR(n - 1);
}

// Iterative:
static int factI(int n) {
    int result = 1;
    for (int i = 2; i <= n; i++) result *= i;
    return result;
}

Both produce the same answer. The recursive version is shorter and arguably matches the mathematical definition more naturally. The iterative version uses constant stack space; the recursive version uses O(n) stack frames and will overflow for very large inputs.

When recursion is the right tool

Recursion is elegant for tree-like or naturally recursive problems: file system traversal, parsing nested structures, processing trees and graphs, divide-and-conquer algorithms (mergesort, quicksort). For straight counting or sequential processing, prefer a loop - it's faster and won't blow the stack. Java doesn't optimize tail calls (yet), so deep recursion can cause StackOverflowError.

A more interesting recursion - Fibonacci

static int fib(int n) {
    if (n < 2) return n;
    return fib(n - 1) + fib(n - 2);
}

// fib(0)=0, fib(1)=1, fib(2)=1, fib(3)=2, fib(4)=3, fib(5)=5, ...

// This naive version recomputes the same values over and over.
// fib(40) makes BILLIONS of calls and takes seconds.
// The iterative version is linear time:

static int fibIter(int n) {
    if (n < 2) return n;
    int a = 0, b = 1;
    for (int i = 2; i <= n; i++) {
        int next = a + b;
        a = b;
        b = next;
    }
    return b;
}

Object-Oriented Programming

Chapter 9 - Classes and Objects

Object-oriented programming is the dominant paradigm in Java. This chapter introduces classes (the blueprints), objects (the instances), fields (the data), and methods (the behavior). Everything in Parts III and IV builds on this foundation.

Classes vs objects - the blueprint analogy

A class is a blueprint or template. An object is an actual instance built from that blueprint. The class Car describes what cars are like; new Car("Toyota") creates one specific car you can drive. You can create as many objects from one class as you want - each is independent, even though they share the same blueprint.

Your first class

public class Person {
    // Fields - data each Person has:
    String name;
    int age;

    // Method - behavior every Person can perform:
    void introduce() {
        System.out.println("Hi, I'm " + name + " and I'm " + age);
    }
}

name and age are fields (also called instance variables). Each Person object gets its own copy. introduce() is a method that uses those fields.

Creating and using objects

Person alice = new Person();
alice.name = "Alice";
alice.age = 30;
alice.introduce();          // Hi, I'm Alice and I'm 30

Person bob = new Person();
bob.name = "Bob";
bob.age = 25;
bob.introduce();            // Hi, I'm Bob and I'm 25

// alice and bob have independent field values.

Three steps every time you use an object:

  • Declare a variable: Person alice;
  • Create with new: alice = new Person();
  • Use via the variable: alice.name = "Alice";

Memory model - stack and heap

Java's memory model has two main regions:

  • The stack - holds local variables and method call frames. Each thread has its own stack. Primitives live here directly.
  • The heap - holds all objects. Shared across threads. Managed by the garbage collector.

When you write Person alice = new Person(), the object is allocated on the heap; the variable on the stack holds a reference (a pointer) to it:

Stack                          Heap
+--------------+              +------------------+
| alice (ref)  |------------->| Person object    |
+--------------+              |   name: null     |
                              |   age: 0         |
                              +------------------+

This is why reference types can be null (no pointer) and why == compares references on objects - it compares the pointers, not what they point at.

Multiple references to one object

Person alice = new Person();
alice.name = "Alice";

Person sameAlice = alice;          // both point to the SAME object
sameAlice.name = "Alicia";

System.out.println(alice.name);    // "Alicia"

Two variables, one object. Any change through one is visible through the other.

Methods and the implicit this

public class Person {
    String name;
    int age;

    void introduce() {
        // These two lines are equivalent:
        System.out.println(name);
        System.out.println(this.name);
    }
}

alice.introduce();   // inside the method, 'this' refers to alice
bob.introduce();     // here, 'this' refers to bob

Most of the time you can omit this. It becomes necessary when a parameter name shadows a field - see Chapter 10.

Methods with parameters and return values

public class Rectangle {
    double width;
    double height;

    double area() {
        return width * height;
    }

    Rectangle scale(double factor) {
        Rectangle scaled = new Rectangle();
        scaled.width = this.width * factor;
        scaled.height = this.height * factor;
        return scaled;
    }
}

Rectangle r = new Rectangle();
r.width = 3; r.height = 4;
r.area();                  // 12.0
Rectangle big = r.scale(2);
big.area();                // 48.0

Default field values

Unlike local variables, fields get default values automatically:

// Defaults for fields:
//   numeric primitives -> 0 or 0.0
//   boolean            -> false
//   char               -> '\u0000'
//   reference types    -> null

Garbage collection

Unlike C, you never explicitly free memory in Java. When an object becomes unreachable, the JVM's garbage collector reclaims its memory:

Person p = new Person();
p = null;                    // object now eligible for GC
GC is your friend, mostly

Garbage collection eliminates entire categories of memory bugs - dangling pointers, double-frees, leaks from forgotten frees. The trade-off is you don't control when collection happens. For low-latency systems, this matters; for 99% of apps, it's fine.

A bank account example

public class Account {
    String owner;
    double balance;

    void deposit(double amount) {
        if (amount <= 0) return;
        balance += amount;
    }

    boolean withdraw(double amount) {
        if (amount <= 0 || amount > balance) return false;
        balance -= amount;
        return true;
    }
}

Account a = new Account();
a.owner = "Alice";
a.balance = 1000;
a.deposit(500);
a.withdraw(200);
// a.balance is now 1300

Setting fields one by one is clumsy. The fix is constructors - Chapter 10.

Chapter 10 - Constructors and the this Keyword

Setting fields one at a time after construction is clumsy. Constructors let you create and initialize an object in one step. This chapter covers constructors, overloading them, chaining them with this(), and the role of the this keyword.

Writing a constructor

public class Person {
    String name;
    int age;

    public Person(String name, int age) {
        this.name = name;
        this.age = age;
    }
}

Person alice = new Person("Alice", 30);

Three things to notice about constructors:

  • Same name as the class - exactly.
  • No return type, not even void.
  • Invoked with new.

The this keyword

Inside the constructor, name means the parameter (it shadows the field). To refer to the field, write this.name:

public Person(String name, int age) {
    this.name = name;     // field = parameter
    this.age = age;
}

// Common bug:
public Person(String name, int age) {
    name = name;          // parameter = parameter (nothing changes!)
    age = age;
}

The default constructor

If you write no constructor, the compiler provides a free no-arg one. Once you write any constructor, the free one disappears:

public class Person {
    public Person(String name, int age) { ... }
}

new Person();             // COMPILE ERROR - no no-arg constructor exists

Overloading constructors

public class Person {
    String name;
    int age;
    String email;

    public Person(String name, int age, String email) {
        this.name = name;
        this.age = age;
        this.email = email;
    }

    public Person(String name, int age) {
        this.name = name;
        this.age = age;
    }

    public Person() {}
}

Chaining constructors with this(...)

The above has duplication. One constructor can call another using this(...):

public class Person {
    String name;
    int age;
    String email;

    public Person(String name, int age, String email) {
        this.name = name;
        this.age = age;
        this.email = email;
    }

    public Person(String name, int age) {
        this(name, age, null);          // delegate to the full one
    }

    public Person() {
        this("Unknown", 0);             // delegate to the two-arg one
    }
}

Rules for this(...):

  • Must be the FIRST statement in the constructor.
  • Each constructor can chain to at most one other.
  • No circular chains.

Validating in constructors

public class Rectangle {
    double width, height;

    public Rectangle(double width, double height) {
        if (width < 0 || height < 0) {
            throw new IllegalArgumentException("Dimensions must be >= 0");
        }
        this.width = width;
        this.height = height;
    }
}

Validate at construction time. It's much easier to reason about an object that can't exist in an invalid state.

Immutable objects via final fields

public class Point {
    final double x;
    final double y;

    public Point(double x, double y) {
        this.x = x;
        this.y = y;
    }
}

Point p = new Point(3, 4);
// p.x = 5;     // COMPILE ERROR - x is final
Why immutability matters

Immutable objects are easier to reason about (their state never surprises you), thread-safe by default, and safe to share freely. Java's String, Integer, and LocalDate are all immutable. Records (Chapter 30) make creating immutable data classes even easier.

    public Circle(double radius) {
        if (radius < 0) throw new IllegalArgumentException("radius < 0");
        this.radius = radius;

Chapter 11 - Inheritance

Inheritance lets one class build on another - reusing fields and methods while adding or customizing behavior. This chapter covers extends, super, constructor chaining with super(), method overriding, and the Object class that every class implicitly extends.

The extends keyword

public class Person {
    String name;
    int age;

    public Person(String name, int age) {
        this.name = name;
        this.age = age;
    }

    void introduce() {
        System.out.println("Hi, I'm " + name);
    }
}

public class Student extends Person {
    String studentId;
    double gpa;

    public Student(String name, int age, String studentId, double gpa) {
        super(name, age);
        this.studentId = studentId;
        this.gpa = gpa;
    }

    void study() {
        System.out.println(name + " is studying.");
    }
}

Student s = new Student("Alice", 20, "S12345", 3.8);
s.introduce();           // inherited from Person
s.study();               // defined in Student

Every Student is a Person. Student inherits all of Person's fields and methods, and adds its own. This is the classic is-a relationship.

super(...) - calling the parent constructor

Constructors are NOT inherited. Each class must define its own. The first thing a subclass constructor must do is initialize the parent part of the object via super(...):

public Student(String name, int age, String studentId, double gpa) {
    super(name, age);          // call Person(name, age)
    this.studentId = studentId;
    this.gpa = gpa;
}

If you don't write super(...) explicitly, the compiler inserts super() - a call to the parent's no-arg constructor. If the parent has no no-arg constructor, you must call super(...) explicitly with arguments.

Method overriding

public class Student extends Person {
    String studentId;

    @Override
    void introduce() {
        System.out.println("Hi, I'm " + name + ", student ID " + studentId);
    }
}

Person p = new Person("Bob", 40);
Student s = new Student("Alice", 20, "S123", 3.8);
p.introduce();    // Hi, I'm Bob
s.introduce();    // Hi, I'm Alice, student ID S123

The @Override annotation

Always use @Override. It tells the compiler 'this should override a parent method' - so if you mistype the name or signature, you get a compile error instead of silently creating a different method.

Calling the parent version with super.method()

public class Student extends Person {
    @Override
    void introduce() {
        super.introduce();          // run Person's version first
        System.out.println("My ID: " + studentId);
    }
}

The Object class

Every Java class extends java.lang.Object, directly or transitively. Object provides a few important methods worth knowing:

  • toString() - returns a String. Default is unhelpful. Almost always worth overriding.
  • equals(Object) - tests equality. Default compares references. Override for value equality.
  • hashCode() - returns an int hash. Must agree with equals.
  • getClass() - runtime class info.

Overriding toString()

public class Person {
    String name;
    int age;

    @Override
    public String toString() {
        return "Person{name='" + name + "', age=" + age + "}";
    }
}

Person p = new Person("Alice", 30);
System.out.println(p);           // Person{name='Alice', age=30}

Overriding equals() and hashCode()

@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof Person)) return false;
    Person other = (Person) o;
    return age == other.age && Objects.equals(name, other.name);
}

@Override
public int hashCode() {
    return Objects.hash(name, age);
}
Equals and hashCode go together

If you override equals, you MUST override hashCode such that equal objects have equal hash codes. Otherwise HashSet, HashMap, and similar will silently misbehave. IDEs generate both at once. We'll see the full contract in Chapter 23.

Preventing inheritance with final

public final class Constants {
    // Cannot be subclassed.
}

public class Util {
    public final void importantMethod() {
        // Subclasses cannot override.
    }
}

String, Integer, and many other JDK classes are final. Use it when subclassing would break the design.

Single inheritance

Java permits one direct superclass per class - no multiple inheritance. This avoids the diamond problem. For multiple inheritance of behavior, Java has interfaces (Chapter 13).

Chapter 12 - Polymorphism

Polymorphism means 'many forms' - the same method call producing different behavior depending on the actual object type. It's one of the four pillars of OOP and the feature that makes inheritance genuinely powerful.

A parent reference, a child object

Animal a = new Dog("Rex");        // declared as Animal, actually a Dog
a.speak();                         // "Rex says: Woof!"

// The method called depends on the actual object type (Dog),
// not the variable's declared type (Animal).
// This is called dynamic dispatch or virtual method invocation.

This is the essence of polymorphism. You can write code that operates on Animal, and at runtime it correctly invokes Dog's or Cat's behavior.

Why this is powerful

public static void makeAllSpeak(List<Animal> animals) {
    for (Animal a : animals) {
        a.speak();
    }
}

List<Animal> zoo = List.of(
    new Dog("Rex"),
    new Cat("Whiskers"),
    new Dog("Buddy")
);
makeAllSpeak(zoo);
// Rex says: Woof!
// Whiskers says: Meow.
// Buddy says: Woof!

makeAllSpeak doesn't know or care what subclasses of Animal exist. Add a new subclass Cow tomorrow, and makeAllSpeak handles it correctly without modification.

Upcasting and downcasting

Upcasting - treating a child as a parent - is implicit and safe:

Dog rex = new Dog("Rex");
Animal a = rex;                   // implicit upcast - always safe
// 'a' can call any Animal method but not Dog-specific ones.

Downcasting - treating a parent as a child - is explicit and can fail:

Animal a = new Dog("Rex");
Dog d = (Dog) a;                  // explicit cast - works because a IS a Dog

Animal a2 = new Cat("Whiskers");
Dog d2 = (Dog) a2;                // ClassCastException at runtime!

instanceof - check before downcasting

// Old style:
if (a instanceof Dog) {
    Dog d = (Dog) a;
    d.fetch();
}

// Java 16+ pattern matching for instanceof - binds in one step:
if (a instanceof Dog d) {
    d.fetch();
}

Virtual dispatch and the @Override contract

When you call a.speak(), the JVM looks up the method on the actual object's class - not the declared type of the variable. This lookup is called virtual dispatch.

class Animal {
    void speak() { System.out.println("..."); }
    void info() { System.out.println("animal info"); }
}

class Dog extends Animal {
    @Override
    void speak() { System.out.println("Woof!"); }
    // info() not overridden - inherited as-is
}

Animal a = new Dog();
a.speak();              // Woof! - Dog's version
a.info();               // animal info - inherited Animal version

Polymorphism with abstract methods (preview)

If a parent has behavior every subclass must define differently, mark the method abstract - which means 'subclasses must override this.' Full coverage in Chapter 13:

public abstract class Shape {
    abstract double area();        // no body - subclasses MUST override

    void describe() {
        System.out.println("Shape with area " + area());
    }
}

public class Circle extends Shape {
    double radius;
    @Override
    double area() { return Math.PI * radius * radius; }
}

Shape s = new Circle();
s.area();         // calls Circle's area

Chapter 13 - Abstract Classes and Interfaces

When you want to define a common API for a family of related classes - without specifying how every method works - you reach for abstract classes or interfaces. This chapter explains both, when to use which, and how interfaces evolved with default methods.

Abstract classes

public abstract class Shape {
    String color;

    public Shape(String color) {
        this.color = color;
    }

    // Concrete method - inherited by all subclasses:
    void describe() {
        System.out.println(color + " shape with area " + area());
    }

    // Abstract method - subclasses MUST provide an implementation:
    abstract double area();
}

// Cannot instantiate an abstract class:
// Shape s = new Shape("red");      // COMPILE ERROR

public class Circle extends Shape {
    double radius;

    public Circle(String color, double radius) {
        super(color);
        this.radius = radius;
    }

    @Override
    double area() { return Math.PI * radius * radius; }
}

Abstract classes:

  • Cannot be instantiated directly - only via concrete subclasses.
  • Can have abstract methods (no body) AND concrete methods (with body).
  • Can have fields and constructors like a normal class.
  • Subclasses must implement all abstract methods (or be abstract themselves).

Interfaces

public interface Drawable {
    void draw();                       // implicitly public abstract
    void erase();
}

public class Circle implements Drawable {
    @Override
    public void draw() {
        System.out.println("Drawing circle");
    }

    @Override
    public void erase() {
        System.out.println("Erasing circle");
    }
}

Interfaces:

  • Pure contracts - originally NO fields and NO method bodies.
  • A class can implement many interfaces (the multiple inheritance Java does allow).
  • All methods are public abstract by default - no need to write it.
  • Any fields declared on an interface are public static final - constants.

Default methods in interfaces (Java 8+)

Java 8 added default methods - methods with a body, in an interface. They were added so the JDK could extend interfaces like List with new methods (like forEach) without breaking the millions of existing classes that implemented those interfaces.

public interface Animal {
    String name();
    String sound();

    // Default method - has a body. Implementers can use as-is or override.
    default void speak() {
        System.out.println(name() + " says: " + sound());
    }
}

public class Dog implements Animal {
    public String name()  { return "Rex"; }
    public String sound() { return "Woof"; }
    // speak() inherited - no need to write it
}

Static methods in interfaces (Java 8+)

public interface Calculator {
    int compute(int x);

    static Calculator squarer() {
        return x -> x * x;
    }
}

Calculator c = Calculator.squarer();

Implementing multiple interfaces

public interface Drawable { void draw(); }
public interface Resizable { void resize(double factor); }

public class Shape implements Drawable, Resizable {
    @Override public void draw() { ... }
    @Override public void resize(double factor) { ... }
}

One class can implement many interfaces. This is how Java does 'multiple inheritance' - of type, not of state.

Abstract class vs interface - when to use which

Use an interface when:

  • You want to specify a capability that unrelated classes might have (Comparable, Closeable).
  • You want a class to be able to implement multiple capabilities.
  • You're designing an API contract with no shared implementation.

Use an abstract class when:

  • You have shared state (fields) and/or shared method implementations.
  • The subclasses are closely related (an 'is-a' hierarchy).
  • You want to enforce a constructor pattern.
Modern default: interface

Since Java 8 added default methods, the lines have blurred. Modern Java code tends to favor interfaces (often with sealed types from Chapter 30 to restrict implementations). Reach for abstract classes when you genuinely need shared state, not just for code reuse.

Chapter 14 - Encapsulation

Encapsulation is the principle of hiding an object's internals behind a public API. You control what code outside the class can see and do. This chapter covers Java's four access modifiers, getters and setters, and immutability as a design strategy.

The four access modifiers

public class Demo {
    public    int p1;     // visible everywhere
    protected int p2;     // same package + subclasses anywhere
              int p3;     // package-private: same package only
    private   int p4;     // this class only
}

Visibility summary (read across):

  • private - same class only.
  • (default) - same package only.
  • protected - same package, plus subclasses in any package.
  • public - everywhere.

Getters and setters

public class Person {
    private String name;
    private int age;

    public Person(String name, int age) {
        this.name = name;
        setAge(age);                    // validate via setter
    }

    public String getName() { return name; }
    public void setName(String name) { this.name = name; }

    public int getAge() { return age; }
    public void setAge(int age) {
        if (age < 0) throw new IllegalArgumentException("age < 0");
        this.age = age;
    }
}

By keeping fields private and exposing getters/setters, you can:

  • Validate input in setters.
  • Compute the value on the fly in getters (e.g., a derived field).
  • Add logging or other side effects.
  • Change the internal representation later without breaking callers.

Immutability - the simplest path

public final class Person {
    private final String name;
    private final int age;

    public Person(String name, int age) {
        this.name = name;
        this.age = age;
    }

    public String getName() { return name; }
    public int getAge() { return age; }
    // No setters. State is fixed at construction.
}

Immutable objects are thread-safe by default, easy to reason about, and safe to share. When in doubt, prefer immutable. Records (Chapter 30) make this even easier.

Don't expose mutable internals

// Bug - exposes the internal list:
public class Team {
    private List<String> members = new ArrayList<>();
    public List<String> getMembers() { return members; }
}

Team t = new Team();
t.getMembers().add("hacker");      // outside code mutates our internal list!

// Fix - return an unmodifiable view, or a copy:
public List<String> getMembers() {
    return Collections.unmodifiableList(members);
}

Chapter 15 - Static, Final, and Nested Classes

This chapter rounds out the class-level vocabulary: the static keyword (things that belong to a class rather than an instance), the final keyword (things that don't change), and nested classes (classes inside other classes).

Static fields

public class Counter {
    static int total = 0;          // shared across all instances
    int id;                        // one per instance

    public Counter() {
        total++;
        id = total;
    }
}

new Counter();  new Counter();  new Counter();
System.out.println(Counter.total);    // 3

Static methods

public class MathUtils {
    public static int square(int n) { return n * n; }
    public static int cube(int n) { return n * n * n; }
}

MathUtils.square(5);              // call without an instance

// Static methods can't access instance state - no 'this':
public class Bad {
    int x;
    static int doIt() {
        return x;                  // COMPILE ERROR - no 'this' available
    }
}
Use static for utility methods

Methods that don't depend on instance state should be static. Math.max, Math.sqrt, Integer.parseInt, Arrays.sort - all static. They're easier to use (no instance needed) and clearer in intent.

Static constants

public class Config {
    public static final int MAX_USERS = 100;
    public static final String DEFAULT_LOCALE = "en_US";
}

Config.MAX_USERS;          // accessed via class name

public static final is the standard Java pattern for constants. Convention: UPPER_SNAKE_CASE.

Static initialization blocks

public class Lookup {
    static final Map<String, Integer> CODES;

    static {
        CODES = new HashMap<>();
        CODES.put("A", 1);
        CODES.put("B", 2);
        CODES.put("C", 3);
    }
}
// The static block runs once, when the class is first loaded.

The final keyword - three uses

final on variables

final int x = 10;
// x = 20;              // ERROR

// On reference variables, final prevents reassignment, NOT mutation:
final List<String> list = new ArrayList<>();
list.add("hello");     // OK - mutating the list
// list = new ArrayList<>();   // ERROR - reassigning the reference

final on methods

public class Base {
    public final void importantStep() {
        // Subclasses cannot override this.
    }
}

final on classes

public final class Immutable {
    // Cannot be subclassed.
}
// String, Integer, and many JDK classes are final.

Nested classes

Java lets you declare a class inside another class. There are four flavors. Most code only ever uses the first one.

1. Static nested class

public class Outer {
    static int count = 0;

    public static class Helper {
        public void doIt() { count++; }      // can access static members
    }
}

Outer.Helper h = new Outer.Helper();
// No Outer instance needed.

Use this when you want to group a helper class inside another class for clarity. It doesn't need access to an enclosing instance.

2. Inner class (non-static nested)

public class Outer {
    int value = 10;

    public class Inner {
        public void doIt() {
            System.out.println(value);    // can access Outer's instance fields
        }
    }
}

Outer outer = new Outer();
Outer.Inner inner = outer.new Inner();    // odd syntax - needs an Outer instance

Each Inner instance has an implicit reference to an Outer instance. Rarely needed in modern code.

3. Local class

public void doStuff() {
    class LocalHelper {           // defined inside a method
        void process() { ... }
    }
    new LocalHelper().process();
}

4. Anonymous class

Runnable r = new Runnable() {
    @Override
    public void run() {
        System.out.println("running");
    }
};

// Modern equivalent with a lambda (Chapter 25):
Runnable r = () -> System.out.println("running");

Anonymous classes were the standard way to pass behavior before Java 8. Now most use cases are better served by lambdas. You'll still see them in older codebases and where you need to implement multiple methods at once.

Chapter 16 — Strings In Depth

Strings are everywhere. Java treats them specially: literal syntax, + concatenation, immutability, and a rich API. This chapter covers what every Java developer needs to know about working with text.

String literals and immutability

String greeting = "Hello";
String upper = greeting.toUpperCase();
System.out.println(greeting);   // Hello — unchanged!
System.out.println(upper);      // HELLO

Strings are immutable: once created, their characters never change. Methods like toUpperCase() return a new String. Immutability makes Strings safe to share across threads and use as keys in HashMaps.

Common String methods

String s = "  Hello, World!  ";

s.length();                // 17
s.trim();                  // "Hello, World!"
s.strip();                 // "Hello, World!" (Unicode-aware, Java 11+)
s.toUpperCase();           // "  HELLO, WORLD!  "
s.toLowerCase();           // "  hello, world!  "
s.contains("World");      // true
s.startsWith("  Hi");     // false
s.endsWith("!  ");        // true
s.indexOf("World");       // 9
s.replace("World", "Java"); // "  Hello, Java!  "
s.substring(2, 7);         // "Hello"
s.split(", ");             // ["  Hello", "World!  "]
s.isEmpty();               // false
s.isBlank();               // false (Java 11+, true if only whitespace)

Comparing strings — never use ==

String a = "hello";
String b = "hello";
String c = new String("hello");

a == b           // true  (string pool — same object)
a == c           // false (different objects, same content)
a.equals(c)      // true  — always use this
a.equalsIgnoreCase("HELLO")   // true
The string pool

Java keeps a 'pool' of string literals. Identical string literals point to the same object — which is why a == b happens to be true above. It's not something to rely on. Always compare strings with .equals().

Concatenation and performance

// + works but creates a new String each time:
String result = "";
for (int i = 0; i < 1000; i++) {
    result += i;   // creates 1000 intermediate Strings — slow
}

// Use StringBuilder for many appends:
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000; i++) {
    sb.append(i);
}
String result2 = sb.toString();

Formatting strings

String name = "Alice";

// Java 15+ formatted method on String
String s2 = "%s is %d years old".formatted(name, age);

// Format specifiers:
//   %s   string
//   %d   integer
//   %f   floating-point (e.g., %.2f for 2 decimal places)
//   %n   platform newline

Chapter 17 — Arrays

Arrays are fixed-size, indexed collections of values of the same type. They're low-level but fast. For variable-size lists you'll usually reach for ArrayList (Chapter 23), but arrays remain everywhere.

Declaring and creating arrays

int[] numbers = new int[5];     // array of 5 ints, all initialized to 0
numbers[0] = 10;
numbers[4] = 50;

// Array literal:
int[] primes = {2, 3, 5, 7, 11};
String[] names = {"Alice", "Bob", "Carol"};

// Length is a field, not a method:
System.out.println(primes.length);   // 5
Indexing starts at 0

An array of length N has valid indices 0 through N-1. Accessing index N or higher throws ArrayIndexOutOfBoundsException.

Iterating arrays

int[] nums = {10, 20, 30, 40};

// Classic for — when you need the index
for (int i = 0; i < nums.length; i++) {
    System.out.println(i + ": " + nums[i]);
}

// Enhanced for — when you just need the values
for (int n : nums) {

Chapter 18 — Exception Handling

Things go wrong: files don't exist, networks fail, users send bad data. Exceptions are Java's mechanism for signaling and handling errors. This chapter covers try/catch, checked vs unchecked, the try-with-resources idiom, and creating your own exception types.

try, catch, finally

try {
    int n = Integer.parseInt("not a number");
} catch (NumberFormatException e) {
    System.out.println("Bad input: " + e.getMessage());
} finally {
    System.out.println("This always runs.");
}

try wraps risky code. catch handles a specific exception type. finally runs no matter what — useful for cleanup.

Multiple catch blocks

try {
    risky();
} catch (FileNotFoundException e) {
    System.out.println("File missing");
} catch (IOException e) {
    System.out.println("I/O error: " + e.getMessage());
} catch (Exception e) {
    System.out.println("Something else broke");
}

// Multi-catch (Java 7+):
try {
    risky();
} catch (FileNotFoundException | NumberFormatException e) {
    System.out.println("Handle both the same way");
}

Checked vs unchecked exceptions

  • Checked exceptions extend Exception (but not RuntimeException). The compiler forces you to either catch them or declare them with throws. Examples: IOException, SQLException.
  • Unchecked exceptions extend RuntimeException. The compiler doesn't force you to handle them. Examples: NullPointerException, IllegalArgumentException, ArrayIndexOutOfBoundsException.
  • Errors extend Error. Don't try to handle these — they signal serious JVM problems like OutOfMemoryError.

Throwing exceptions

public void setAge(int age) {
    if (age < 0) {
        throw new IllegalArgumentException("age must be non-negative, got " + age);
    }
    this.age = age;
}

The throws clause

// If you call a method that throws a checked exception,
// you must either catch it or declare it:

public String readFile(String path) throws IOException {
    return Files.readString(Path.of(path));
    // We're not catching IOException, so we declare it
    // and let the caller deal with it.
}

try-with-resources

If a resource needs to be closed (files, network sockets, database connections), use try-with-resources. It calls close() automatically, even on exception.

try (var reader = new BufferedReader(new FileReader("data.txt"))) {
    String line;
    while ((line = reader.readLine()) != null) {
        System.out.println(line);
    }
}
// reader.close() is called automatically when the try block exits

Custom exceptions

// Checked exception — extends Exception
public class InsufficientFundsException extends Exception {
    public InsufficientFundsException(String message) {
        super(message);
    }
}

// Unchecked — extends RuntimeException
public class InvalidUserStateException extends RuntimeException {
    public InvalidUserStateException(String message) {
        super(message);
    }
}
Don't swallow exceptions

An empty catch block — catch (Exception e) { } — is almost always wrong. At minimum log the exception. Hiding errors makes debugging enormously harder.

Checked exceptions are part of a method's signature — callers MUST handle them or re-throw. Use them for recoverable conditions you want callers to think about (e.g., a file not being found). Chapter 19 — Wrapper Classes and Autoboxing

Primitives are fast, but you can't put them in a List or HashMap — those need objects. Wrapper classes bridge the gap. This chapter covers the wrappers, autoboxing, and the performance gotcha you should know about.

The eight wrapper classes

  • byte → Byte
  • short → Short
  • int → Integer
  • long → Long
  • float → Float
  • double → Double
  • boolean → Boolean
  • char → Character

Autoboxing and unboxing

// Autoboxing: primitive → wrapper (automatic)
Integer boxed = 42;          // really Integer.valueOf(42)

// Unboxing: wrapper → primitive (automatic)
int unboxed = boxed;         // really boxed.intValue()

// In collections — only objects allowed:
List<Integer> numbers = new ArrayList<>();
numbers.add(10);              // autoboxes int to Integer
int first = numbers.get(0);   // unboxes Integer to int

Useful methods on wrappers

Integer.parseInt("42");                  // 42
Integer.toBinaryString(10);              // "1010"
Integer.toHexString(255);                // "ff"
Integer.max(3, 5);                       // 5
Integer.MAX_VALUE;                       // 2147483647
Integer.MIN_VALUE;                       // -2147483648

Double.parseDouble("3.14");             // 3.14
Double.isNaN(0.0 / 0.0);                 // true
Double.isInfinite(1.0 / 0.0);            // true

Boolean.parseBoolean("true");           // true
Character.isDigit('5');                  // true
Character.isLetter('A');                 // true
Character.toUpperCase('a');              // 'A'

Equality of wrappers — beware ==

Integer a = 100;
Integer b = 100;
System.out.println(a == b);        // true

Integer c = 200;
Integer d = 200;
System.out.println(c == d);        // false (!)

// Always use .equals() for wrapper equality:
System.out.println(c.equals(d));   // true
The cache

Java caches Integer values from -128 to 127. Small autoboxed Integers are reused, so == happens to work for them. Outside that range, every autoboxed value is a new object. Always use .equals() or unbox to primitives for numeric comparison.

Performance considerations

Autoboxing is convenient but has a cost — wrapper objects use much more memory than primitives, and create garbage to collect. In hot loops or memory-sensitive code, prefer primitive types and primitive specializations:

// Slow — boxes every int:
List<Integer> list = new ArrayList<>();
for (int i = 0; i < 1_000_000; i++) list.add(i);

// Fast — pure primitive array:
int[] arr = new int[1_000_000];
for (int i = 0; i < 1_000_000; i++) arr[i] = i;

// Or use IntStream (Chapter 28):
int sum = IntStream.range(0, 1_000_000).sum();

Chapter 20 — The Date and Time API

Java's old date API (Date, Calendar) was famously awful. Java 8 replaced it with the java.time package — immutable, type-safe, thread-safe, and actually pleasant to use. This is what you should always use.

The core types

  • LocalDate — a date without time or timezone (2026-05-15).
  • LocalTime — a time without date or timezone (14:30:00).
  • LocalDateTime — date + time, no timezone.
  • ZonedDateTime — date + time + timezone.
  • Instant — a point on the timeline (Unix-style nanosecond precision).
  • Duration — an amount of time (seconds + nanos).
  • Period — an amount of date (years, months, days).

Creating dates and times

Instant instant = Instant.now();   // typically used for timestamps

Chapter 21 — Enums

Enums represent a fixed set of constants. Use them whenever a variable can only hold one of a small, predefined set of values (days of the week, order statuses, log levels). They're far safer than integer or String constants.

Defining and using an enum

public enum Day {
    MONDAY, TUESDAY, WEDNESDAY, THURSDAY,
    FRIDAY, SATURDAY, SUNDAY
}

Day today = Day.WEDNESDAY;

if (today == Day.WEDNESDAY) {   // == is fine for enums
    System.out.println("Hump day!");
}

Enums with switch

Day day = Day.FRIDAY;
String mood = switch (day) {
    case MONDAY -> "ugh";
    case FRIDAY -> "yes!";
    case SATURDAY, SUNDAY -> "weekend";
    default -> "meh";
};

Enum methods

Day.values();              // array of all values
Day.valueOf("FRIDAY");    // looks up by name (throws if not found)
Day.MONDAY.name();         // "MONDAY"
Day.MONDAY.ordinal();      // 0 — position in declaration order

// Iterate:
for (Day d : Day.values()) {
    System.out.println(d);
}

Enums with fields and methods

Enums are full classes. They can have fields, constructors, and methods.

public enum Planet {
    MERCURY(3.303e+23, 2.4397e6),
    VENUS  (4.869e+24, 6.0518e6),
    EARTH  (5.976e+24, 6.37814e6);

    private final double mass;     // in kilograms
    private final double radius;   // in meters

    Planet(double mass, double radius) {
        this.mass = mass;
        this.radius = radius;
    }

        final double G = 6.67300E-11;

    },

Chapter 22 — Generics

Generics let you write classes and methods that work with any type, while keeping compile-time type safety. They're how List<String> and Map<String, User> work. Understanding them is essential before tackling the Collections framework.

The problem generics solve

// Pre-generics (Java 1.4 and earlier) — raw List:
List names = new ArrayList();
names.add("Alice");
names.add(42);                  // compiles fine — uh oh
String s = (String) names.get(1);   // ClassCastException at runtime!

// With generics:
List<String> names = new ArrayList<>();
names.add("Alice");
names.add(42);                  // ← compile error, caught immediately
String s = names.get(0);        // no cast needed

Writing a generic class

public class Box<T> {
    private T value;

    public Box(T value) {
        this.value = value;
    }

    public T get() {
        return value;
    }

    public void set(T value) {
        this.value = value;
    }
}

Box<String> sb = new Box<>("hello");
String s = sb.get();

Box<Integer> ib = new Box<>(42);
int n = ib.get();

T is a type parameter — a placeholder. You can name it anything, but conventions are: T (type), E (element), K (key), V (value), R (result), N (number).

The diamond operator

// Pre-Java 7 — verbose:
Map<String, List<Integer>> map = new HashMap<String, List<Integer>>();

// Java 7+ — diamond infers the right side:
Map<String, List<Integer>> map = new HashMap<>();

Generic methods

public static <T> T firstOrDefault(List<T> list, T defaultValue) {
    if (list.isEmpty()) return defaultValue;
    return list.get(0);
}

String first = firstOrDefault(List.of("a", "b"), "none");   // "a"
Integer n = firstOrDefault(List.<Integer>of(), -1);              // -1

Bounded type parameters

Restrict a type parameter to a subclass of something:

// T must extend Number — so we can call .doubleValue()
public static <T extends Number> double sum(List<T> list) {
    double total = 0;
    for (T n : list) total += n.doubleValue();
    return total;
}

sum(List.of(1, 2, 3));         // OK — Integer extends Number
sum(List.of(1.5, 2.5));        // OK — Double extends Number
// sum(List.of("a", "b")); ← compile error — String is not a Number

Wildcards

A wildcard ? means "some unknown type." It's used when you don't care what specific type a generic holds:

// Accepts ANY List, regardless of element type:
public static void printAll(List<?> list) {
    for (Object o : list) {
        System.out.println(o);
    }
}

// Upper bound — anything Number or a subtype:
public static double sumOf(List<? extends Number> list) {
    double total = 0;
    for (Number n : list) total += n.doubleValue();
    return total;
}

// Lower bound — anything Integer or a supertype:
public static void addInts(List<? super Integer> list) {
    list.add(1);
    list.add(2);
}
PECS — Producer Extends, Consumer Super

If a parameter produces values you'll read, use ? extends T. If it consumes values you'll write, use ? super T. This rule helps you choose the right wildcard.

Type erasure — the catch

Generics exist only at compile time. At runtime, List<String> and List<Integer> are both just List. This is called erasure. Consequences:

  • You can't write new T() — the JVM doesn't know what T is at runtime.
  • You can't write new T[10] for the same reason.
  • You can't use instanceof List<String> — just instanceof List.
  • Method overloading can't differ only in generic parameters (List<String> vs List<Integer> have the same erased signature).

Chapter 23 — The Collections Framework

Java's Collections framework gives you ready-made data structures: lists, sets, queues, and maps. This is one of the most-used parts of the standard library. Knowing which collection to pick when is a daily skill.

The big picture

Iterable<T>
  └── Collection<T>
        ├── List<T>     — ordered, allows duplicates, indexable
        ├── Set<T>      — no duplicates, generally unordered
        └── Queue<T>    — designed for processing elements in order

Map<K, V>                — key → value pairs (NOT a Collection)

List — ordered, indexed, allows duplicates

List<String> list = new ArrayList<>();
list.add("Alice");
list.add("Bob");
list.add("Alice");          // duplicates OK

list.size();                // 3
list.get(0);                // "Alice"
list.set(0, "Anna");        // replace index 0
list.remove(1);             // remove by index
list.remove("Anna");        // remove by value (first occurrence)
list.contains("Bob");       // true
list.indexOf("Bob");        // index of first occurrence
list.isEmpty();             // false
list.clear();               // empty the list

// Immutable list (Java 9+):
List<Integer> immutable = List.of(1, 2, 3);   // cannot add/remove/modify

Two main List implementations:

  • ArrayList — backed by a resizable array. Fast random access, fast appends. Use this by default.
  • LinkedList — doubly-linked list. Fast inserts/removes at the ends, but slow random access. Rarely the right choice; ArrayList beats it in most real workloads.

Set — no duplicates

Set<String> set = new HashSet<>();
set.add("apple");
set.add("banana");
set.add("apple");          // ignored — already present
set.size();                 // 2
set.contains("apple");     // true

// Set.of for immutable:
Set<Integer> primes = Set.of(2, 3, 5, 7);
  • HashSet — fast (O(1)) add, remove, contains. No order guarantee. Default choice.
  • LinkedHashSet — like HashSet but preserves insertion order.
  • TreeSet — keeps elements sorted (natural order or by a Comparator). O(log n) operations.

Map — key/value pairs

Map<String, Integer> ages = new HashMap<>();
ages.put("Alice", 30);
ages.put("Bob", 25);

ages.get("Alice");                  // 30
ages.get("Carol");                  // null (no such key)
ages.getOrDefault("Carol", -1);     // -1
ages.containsKey("Alice");          // true
ages.containsValue(30);             // true
ages.remove("Bob");
ages.size();                        // 1

// Iterate:
for (Map.Entry<String, Integer> e : ages.entrySet()) {
    System.out.println(e.getKey() + " -> " + e.getValue());
}

// Or just keys / just values:
for (String name : ages.keySet()) { ... }
for (int age : ages.values()) { ... }

// Immutable (Java 9+):
Map<String, Integer> m = Map.of("a", 1, "b", 2);
  • HashMap — O(1) operations. No order guarantee. Default choice.
  • LinkedHashMap — preserves insertion order.
  • TreeMap — keys kept sorted. O(log n) operations.

Queue and Deque

Queue<String> q = new LinkedList<>();
q.offer("a");          // add to tail
q.offer("b");
q.poll();              // "a" — remove from head
q.peek();              // "b" — look without removing

Deque<String> d = new ArrayDeque<>();   // double-ended queue
d.push("x");          // add to head (stack-style)
d.push("y");
d.pop();               // "y"

The Collections utility class

import java.util.Collections;

List<Integer> nums = new ArrayList<>(List.of(3, 1, 4, 1, 5));
Collections.sort(nums);              // [1, 1, 3, 4, 5]
Collections.reverse(nums);           // [5, 4, 3, 1, 1]
Collections.shuffle(nums);           // random order
Collections.max(nums);               // 5
Collections.min(nums);               // 1
Collections.frequency(nums, 1);      // 2 (occurrences of 1)

Choosing the right collection

  • Need ordered, indexable, duplicates? → ArrayList
  • Need fast lookup by content, no duplicates? → HashSet
  • Need key → value mapping? → HashMap
  • Need insertion-order preserved with set/map semantics? → LinkedHashSet / LinkedHashMap
  • Need sorted iteration? → TreeSet / TreeMap
  • Need FIFO/LIFO queue? → ArrayDeque
Equals and hashCode contract

HashSet and HashMap use hashCode() to place objects in buckets, then equals() to disambiguate. If you put your own class into one of these, you MUST override both — and they must agree: equal objects MUST have equal hash codes. IDEs generate both for you.

The default implementations from Object use the object's memory address. Two of your objects with identical field values will be considered unequal and hash to different buckets. Chapter 24 — Iterators and Comparators

Two related concepts that show up everywhere in the Collections framework: Iterator (how you walk through a collection) and Comparator (how you tell Java to compare two things for ordering).

The Iterator interface

List<String> names = new ArrayList<>(List.of("Alice", "Bob", "Carol"));

Iterator<String> it = names.iterator();
while (it.hasNext()) {
    String name = it.next();
    System.out.println(name);
}

Most of the time you'll just use the enhanced for loop, which uses an Iterator under the hood. But for safe removal during iteration, you need the Iterator directly:

// ❌ ConcurrentModificationException:
for (String s : names) {
    if (s.startsWith("A")) names.remove(s);
}

// ✓ Safe via Iterator:
Iterator<String> it = names.iterator();
while (it.hasNext()) {
    if (it.next().startsWith("A")) {
        it.remove();
    }
}

// ✓ Or even better (Java 8+):
names.removeIf(s -> s.startsWith("A"));

Comparable — natural ordering

When a class implements Comparable<T>, it defines its natural order. Strings sort alphabetically, Integers numerically, dates chronologically — all via Comparable.

public class Person implements Comparable<Person> {
    String name;
    int age;

    public Person(String name, int age) {
        this.name = name;
        this.age = age;
    }

    @Override
    public int compareTo(Person other) {
        // negative: this < other
        // zero:     equal
        // positive: this > other
        return Integer.compare(this.age, other.age);
    }
}

List<Person> people = new ArrayList<>(List.of(
    new Person("Bob", 30),
    new Person("Alice", 25)
));
Collections.sort(people);   // sorts by age — Alice first

Comparator — custom orderings

A class has only one natural order. For other orderings — by name, by salary, reverse, multi-key — use a Comparator:

// Anonymous class (old style):
people.sort(new Comparator<Person>() {
    @Override
    public int compare(Person a, Person b) {
        return a.name.compareTo(b.name);
    }
});

// Lambda (modern):
people.sort((a, b) -> a.name.compareTo(b.name));

// Even better — Comparator factory methods:
people.sort(Comparator.comparing(p -> p.name));
people.sort(Comparator.comparingInt(p -> p.age));

Composing comparators

// Sort by name, then by age as tiebreaker, descending:
Comparator<Person> byNameThenAge = Comparator
    .comparing((Person p) -> p.name)
    .thenComparingInt(p -> p.age)
    .reversed();

people.sort(byNameThenAge);

Sorting with a Comparator

// Sort a List:
names.sort(Comparator.naturalOrder());
names.sort(Comparator.reverseOrder());

// Sort an array:
Integer[] nums = {3, 1, 4, 1, 5};
Arrays.sort(nums, Comparator.reverseOrder());

// TreeSet / TreeMap with custom order:
TreeSet<String> caseInsensitive = new TreeSet<>(String.CASE_INSENSITIVE_ORDER);
Comparable vs Comparator

Comparable goes on the class you control — it defines THE natural order. Comparator is a separate object — pass one in when you want a different order at a specific call site. You can have many Comparators for one Comparable class.

Chapter 25 — Lambda Expressions

Java 8 introduced lambdas — compact syntax for anonymous functions. They unlocked a whole new style of writing Java: passing behavior around like data. This chapter starts a four-chapter arc on functional programming.

From anonymous class to lambda

// Java 7 — anonymous class:
Runnable task = new Runnable() {
    @Override
    public void run() {
        System.out.println("Running");
    }
};

// Java 8 — lambda:
Runnable task = () -> System.out.println("Running");

Lambda syntax

// No parameters:
() -> System.out.println("Hi")

// One parameter — parentheses optional:
x -> x * 2
(x) -> x * 2          // equivalent

// Multiple parameters — parentheses required:
(x, y) -> x + y

// With explicit types:
(int x, int y) -> x + y

// Multi-line body — use { } and return:
(x, y) -> {
    int sum = x + y;
    return sum * 2;
}

Lambdas implement functional interfaces

A lambda is shorthand for implementing a functional interface — an interface with exactly one abstract method. The next chapter goes deep on functional interfaces. For now, the key insight is that a lambda's type depends on the context where it's used:

@FunctionalInterface
interface Greeting {
    String greet(String name);
}

Greeting friendly = name -> "Hello, " + name + "!";
Greeting formal   = name -> "Good day, " + name + ".";

System.out.println(friendly.greet("Alice"));   // Hello, Alice!
System.out.println(formal.greet("Alice"));     // Good day, Alice.

Capturing variables

int multiplier = 3;
Function<Integer, Integer> triple = x -> x * multiplier;
// Lambda 'captures' multiplier from the enclosing scope.

triple.apply(10);   // 30
Effectively final

A lambda can only capture variables that are final or effectively final (unchanged after assignment). The compiler enforces this. Reason: lambdas may be stored, passed around, and run later — possibly on another thread — and Java's designers didn't want lambdas to see shifting local state.

Lambdas everywhere in the JDK

List<String> names = new ArrayList<>(List.of("Bob", "Alice", "Carol"));

// forEach — Consumer<T>
names.forEach(name -> System.out.println(name));

// sort — Comparator<T>
names.sort((a, b) -> a.length() - b.length());

// removeIf — Predicate<T>
names.removeIf(name -> name.startsWith("A"));

// replaceAll — UnaryOperator<T>
names.replaceAll(name -> name.toUpperCase());

When NOT to use a lambda

Lambdas are great for short, focused behavior. They get unwieldy past a few lines. If the body spans many lines or needs comments to explain, extract it to a named method or class.

Chapter 26 — Functional Interfaces

A functional interface is an interface with exactly one abstract method. Lambdas and method references can implement them. The java.util.function package gives you 40+ ready-made functional interfaces — but you really only need to know about six.

The big six

  • Function<T, R> — takes a T, returns an R. R apply(T t)
  • Predicate<T> — takes a T, returns boolean. boolean test(T t)
  • Consumer<T> — takes a T, returns nothing. void accept(T t)
  • Supplier<T> — takes nothing, returns a T. T get()
  • UnaryOperator<T> — Function from T to T. T apply(T t)
  • BiFunction<T, U, R> — takes T and U, returns R. R apply(T t, U u)

Function — transforming a value

Function<String, Integer> length = s -> s.length();
length.apply("hello");   // 5

Function<Integer, String> describe = n -> "got " + n;

// Composition:
Function<String, String> pipeline = length.andThen(describe);
pipeline.apply("hello");   // "got 5"

Predicate — testing a condition

Predicate<Integer> isPositive = n -> n > 0;
Predicate<Integer> isEven     = n -> n % 2 == 0;

isPositive.test(5);                              // true

// Combining predicates:
Predicate<Integer> positiveAndEven = isPositive.and(isEven);
Predicate<Integer> positiveOrEven  = isPositive.or(isEven);
Predicate<Integer> notPositive     = isPositive.negate();

// Streams use predicates everywhere:
List.of(-2, -1, 0, 1, 2, 3).stream()
    .filter(positiveAndEven)
    .toList();   // [2]

Consumer — doing something with a value

Consumer<String> print = s -> System.out.println(s);
print.accept("hello");

List.of("a", "b", "c").forEach(print);

Supplier — producing a value on demand

Supplier<LocalDate> today = () -> LocalDate.now();
today.get();

// Useful for lazy / on-demand creation:
static <T> T orElseGet(T value, Supplier<T> fallback) {
    return value != null ? value : fallback.get();
}
// fallback.get() is only called if value is null —
// avoiding expensive computation when not needed.

Primitive specializations

To avoid autoboxing in hot code, there are primitive versions: IntFunction, IntPredicate, IntConsumer, IntSupplier, and similar for Long and Double.

IntPredicate isEven = n -> n % 2 == 0;
// No boxing — n stays a primitive int the whole way.

IntFunction<String> describe = n -> "value: " + n;
// Takes int, returns String.

Writing your own functional interface

@FunctionalInterface
public interface TriFunction<A, B, C, R> {
    R apply(A a, B b, C c);
}

TriFunction<Integer, Integer, Integer, Integer> sumThree =
    (a, b, c) -> a + b + c;

sumThree.apply(1, 2, 3);   // 6
@FunctionalInterface

The annotation is optional but recommended. It tells the compiler to verify there's exactly one abstract method — protecting your interface from accidental modifications that would break lambda compatibility.

Chapter 27 — Method References

A method reference is shorthand for a lambda that just calls an existing method. Same behavior, terser syntax. Once you know the four forms, you'll recognize them everywhere in modern Java code.

The :: operator

// Lambda that delegates to an existing method:
names.forEach(s -> System.out.println(s));

// Same thing as a method reference:
names.forEach(System.out::println);

Four forms of method reference

1. Static method reference

// Lambda:        s -> Integer.parseInt(s)
// Method ref:    Integer::parseInt

Function<String, Integer> parse = Integer::parseInt;
parse.apply("42");   // 42

2. Reference to a method on a specific instance

// Lambda:        s -> System.out.println(s)
// Method ref:    System.out::println

// 'System.out' is the specific instance.
Consumer<String> printer = System.out::println;
printer.accept("hello");

3. Reference to an instance method of an arbitrary instance

// Lambda:        s -> s.toUpperCase()
// Method ref:    String::toUpperCase

// Each input becomes the instance the method is called on.
Function<String, String> upper = String::toUpperCase;
upper.apply("hi");   // "HI"

// And with parameters:
// Lambda:        (a, b) -> a.compareTo(b)
// Method ref:    String::compareTo

Comparator<String> cmp = String::compareTo;

4. Constructor reference

// Lambda:        () -> new ArrayList<>()
// Constructor:   ArrayList::new

Supplier<List<String>> listFactory = ArrayList::new;
List<String> empty = listFactory.get();

// With argument:
// Lambda:        n -> new ArrayList<>(n)
// Constructor:   ArrayList::new   (compiler picks the right overload)

Function<Integer, List<String>> sizedListFactory = ArrayList::new;
List<String> capacity10 = sizedListFactory.apply(10);

When method references shine

List<String> names = List.of("alice", "bob", "carol");

// Lambda chain — readable but noisy:
names.stream()
     .map(s -> s.toUpperCase())
     .filter(s -> s.startsWith("A"))
     .forEach(s -> System.out.println(s));

// Method references — cleaner:
names.stream()
     .map(String::toUpperCase)
     .filter(s -> s.startsWith("A"))   // can't ref this — has a literal arg
     .forEach(System.out::println);
When to prefer a lambda

Method references win when there's a 1:1 mapping to an existing method. If your lambda passes constants, reorders arguments, or does anything extra, stick with the lambda — clarity beats brevity.

Chapter 28 — The Stream API

Streams are the headline feature of Java 8 functional programming. They let you express data transformations as a pipeline of operations — filter, map, reduce, collect — instead of nested loops. This is the chapter where everything from Chapters 25-27 comes together.

Creating a Stream

// From a Collection:
Stream<String> s1 = List.of("a", "b", "c").stream();

// From values:
Stream<Integer> s2 = Stream.of(1, 2, 3);

// From an array:
Stream<String> s3 = Arrays.stream(new String[]{"a", "b"});

// Infinite stream — must be limited:
Stream<Integer> s4 = Stream.iterate(1, n -> n + 1).limit(10);

// Range of ints (primitive stream):
IntStream s5 = IntStream.range(0, 10);        // 0..9
IntStream s6 = IntStream.rangeClosed(1, 10);  // 1..10

Intermediate operations — return another Stream

List<String> names = List.of("Alice", "Bob", "Carol", "Dave");

names.stream()
     .filter(s -> s.length() > 3)        // keep matching elements
     .map(String::toUpperCase)            // transform each
     .sorted()                            // sort
     .distinct()                          // remove duplicates
     .limit(2)                            // first N
     .skip(1)                             // skip first N
     .peek(System.out::println);          // debug — see what passes through

// Nothing has happened yet! Intermediate ops are LAZY.
// The pipeline only runs when a terminal op is added.

Terminal operations — produce a result

// Collect into a List:
List<String> result = names.stream()
    .filter(s -> s.length() > 3)
    .toList();   // Java 16+ — immutable list

// Count:
long n = names.stream().filter(s -> s.startsWith("A")).count();

// First match:
Optional<String> first = names.stream()
    .filter(s -> s.startsWith("C"))
    .findFirst();   // Optional<String>

// Match tests:
boolean anyA = names.stream().anyMatch(s -> s.startsWith("A"));
boolean allLong = names.stream().allMatch(s -> s.length() > 2);
boolean noneX = names.stream().noneMatch(s -> s.startsWith("X"));

// forEach:
names.stream().forEach(System.out::println);

// Reduce — fold the stream to a single value:
int sum = Stream.of(1, 2, 3, 4).reduce(0, Integer::sum);   // 10
String joined = names.stream().reduce("", (a, b) -> a + b);

Collectors — flexible collection-building

import static java.util.stream.Collectors.*;

// To a List, Set, or specific collection:
names.stream().collect(toList());           // mutable list
names.stream().collect(toSet());
names.stream().collect(toCollection(TreeSet::new));

// To a Map: <key extracted from each element, value extracted from each>
Map<String, Integer> byLength = names.stream()
    .collect(toMap(s -> s, String::length));
// {"Alice" -> 5, "Bob" -> 3, ...}

// Joining strings:
String csv = names.stream().collect(joining(", "));
String wrapped = names.stream().collect(joining(", ", "[", "]"));

// Grouping:
Map<Integer, List<String>> byLen = names.stream()
    .collect(groupingBy(String::length));
// {3 -> [Bob], 4 -> [Dave], 5 -> [Alice, Carol]}

// Counting in groups:
Map<Integer, Long> counts = names.stream()
    .collect(groupingBy(String::length, counting()));

// Partitioning (two buckets — true/false):
Map<Boolean, List<String>> partitioned = names.stream()
    .collect(partitioningBy(s -> s.length() > 3));

Primitive streams

// IntStream, LongStream, DoubleStream — no boxing, plus numeric ops:
IntStream.rangeClosed(1, 100).sum();           // 5050
IntStream.rangeClosed(1, 100).average();       // OptionalDouble[50.5]
IntStream.rangeClosed(1, 100).max();           // OptionalInt[100]
IntStream.rangeClosed(1, 100).summaryStatistics();
// IntSummaryStatistics — count, sum, min, avg, max all at once

// Convert between:
Stream<Integer> boxed = IntStream.range(0, 10).boxed();
IntStream unboxed = Stream.of(1, 2, 3).mapToInt(Integer::intValue);

Putting it together — a realistic example

record Order(String customer, double amount, String status) {}

List<Order> orders = List.of(
    new Order("Alice", 100, "paid"),
    new Order("Bob",   50, "pending"),
    new Order("Alice", 200, "paid"),
    new Order("Carol", 75, "paid"),
    new Order("Bob",  300, "paid")
);

// Total revenue from paid orders, by customer:
Map<String, Double> revenue = orders.stream()
    .filter(o -> o.status().equals("paid"))
    .collect(groupingBy(
        Order::customer,
        summingDouble(Order::amount)
    ));
// {"Alice" -> 300.0, "Bob" -> 300.0, "Carol" -> 75.0}
Streams are one-shot

A stream can only be traversed once. Calling a terminal operation 'consumes' it. If you need to process the same data multiple ways, call .stream() on the underlying collection again, or store the intermediate result.

Parallel streams

long sum = IntStream.rangeClosed(1, 1_000_000)
    .parallel()
    .sum();

// Spreads work across CPU cores via ForkJoinPool.
// Only worth it for large datasets and CPU-bound operations.
// Measure before assuming parallel is faster — overhead can dominate.

Chapter 29 — Multithreading and Concurrency

Java has supported threads since day one. This chapter covers the classic threading model — Thread, Runnable, synchronized — then jumps to modern, higher-level tools like ExecutorService and CompletableFuture that you should usually reach for first.

Creating a thread

// Implement Runnable (preferred):
Runnable task = () -> {
    System.out.println("Hi from " + Thread.currentThread().getName());
};

Thread t = new Thread(task, "worker-1");
t.start();          // runs concurrently
t.join();           // wait for it to finish

// Less common: extend Thread:
class Worker extends Thread {
    @Override public void run() {
        // ...
    }
}
start() vs run()

Calling t.start() spawns a new thread. Calling t.run() directly just invokes the method on the current thread — no concurrency. This is a classic beginner trap.

The shared-state problem

class Counter {
    int value = 0;
    void inc() { value++; }
}

// 1000 threads each increment 1000 times — total should be 1,000,000:
Counter c = new Counter();
List<Thread> threads = new ArrayList<>();
for (int i = 0; i < 1000; i++) {
    Thread t = new Thread(() -> {
        for (int j = 0; j < 1000; j++) c.inc();
    });
    threads.add(t);
    t.start();
}
for (Thread t : threads) t.join();
System.out.println(c.value);   // almost certainly LESS than 1,000,000

value++ looks atomic but isn't — it reads, increments, and writes. Two threads can interleave and lose updates. This is a race condition.

Fixing it: synchronized

class Counter {
    int value = 0;
    synchronized void inc() { value++; }
}

// Now only one thread can be inside inc() at a time.

Atomic classes — lock-free

import java.util.concurrent.atomic.AtomicInteger;

class Counter {
    AtomicInteger value = new AtomicInteger(0);
    void inc() { value.incrementAndGet(); }
    int get() { return value.get(); }
}

// Faster than synchronized for simple cases — uses CPU atomic instructions.

ExecutorService — don't manage threads yourself

import java.util.concurrent.*;

ExecutorService pool = Executors.newFixedThreadPool(4);

for (int i = 0; i < 10; i++) {
    int taskId = i;
    pool.submit(() -> {
        System.out.println("task " + taskId + " on "
                           + Thread.currentThread().getName());
    });
}

pool.shutdown();   // stop accepting new tasks
pool.awaitTermination(1, TimeUnit.MINUTES);   // wait for pending

Returning values: Future and Callable

Future<Integer> future = pool.submit(() -> {
    Thread.sleep(1000);
    return 42;
});

int result = future.get();   // blocks until the task is done

// Or specify a timeout:
future.get(5, TimeUnit.SECONDS);

CompletableFuture — composable async

CompletableFuture<String> task = CompletableFuture
    .supplyAsync(() -> fetchData())             // run async
    .thenApply(data -> data.toUpperCase())      // transform result
    .thenApply(s -> "processed: " + s)
    .exceptionally(ex -> "error: " + ex.getMessage());

String result = task.get();   // blocks for final result

// Combining results:
CompletableFuture<String> a = CompletableFuture.supplyAsync(() -> "hello");
CompletableFuture<String> b = CompletableFuture.supplyAsync(() -> "world");
CompletableFuture<String> combined = a.thenCombine(b, (x, y) -> x + " " + y);

Concurrent collections

// Thread-safe replacements for HashMap, ArrayList, etc.:
import java.util.concurrent.*;

ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();
map.put("a", 1);
map.compute("a", (k, v) -> (v == null ? 0 : v) + 1);   // atomic update

CopyOnWriteArrayList<String> list = new CopyOnWriteArrayList<>();
// Reads are unsynchronized; writes copy the underlying array.
// Good when reads vastly outnumber writes.
Modern advice

Prefer high-level tools: ExecutorService, CompletableFuture, parallel streams, concurrent collections. Reach for raw Thread and synchronized only when you have a specific reason. Concurrent code is notoriously hard to get right; build on tested abstractions.

Exercise 29.2

Why does value++ become a race condition under concurrent access, even though it looks like one statement?

Solution to Exercise 29.2

value++ compiles to three steps: read the current value, increment it, write it back. If two threads execute these steps interleaved, both might read the same starting value, both compute the same new value, and both write it — losing one update. The expression is one statement in source but three operations at the machine level. Fix with synchronized, AtomicInteger, or higher-level constructs.

Chapter 30 — Modern Java Features (17–25)

Java's release cadence has been steady — a new version every six months. This chapter is a tour of the features added since Java 17 that you'll actually use day to day: records, sealed classes, pattern matching, text blocks, switch expressions, var, and more.

Records — concise data classes (Java 16+)

// Before — boilerplate hell:
public class Point {
    private final int x;
    private final int y;

    public Point(int x, int y) { this.x = x; this.y = y; }
    public int x() { return x; }
    public int y() { return y; }

    @Override public boolean equals(Object o) { /* boilerplate */ }
    @Override public int hashCode() { /* boilerplate */ }
    @Override public String toString() { /* boilerplate */ }
}

// After — one line:
public record Point(int x, int y) {}

Point p = new Point(3, 4);
p.x();                    // 3
p.y();                    // 4
p.toString();             // "Point[x=3, y=4]"
p.equals(new Point(3,4)); // true

Records are immutable. They auto-generate constructor, accessors, equals, hashCode, and toString. You can add your own methods or static factories.

public record Range(int min, int max) {
    // Compact constructor — validates input
    public Range {
        if (min > max) throw new IllegalArgumentException();
    }

    // Extra methods are fine
    public int size() { return max - min; }
}

Sealed classes (Java 17+)

A sealed class restricts which classes can extend it. Combined with pattern matching, you get exhaustive type hierarchies the compiler can verify.

public sealed interface Shape permits Circle, Square, Triangle {}

public record Circle(double radius) implements Shape {}
public record Square(double side) implements Shape {}
public record Triangle(double base, double height) implements Shape {}

// Now nobody outside this list can implement Shape.
// switch on Shape becomes exhaustive — no default needed:
double area = switch (s) {
    case Circle c   -> Math.PI * c.radius() * c.radius();
    case Square sq  -> sq.side() * sq.side();
    case Triangle t -> 0.5 * t.base() * t.height();
};

// If you add Hexagon to permits, the compiler will tell you
// EVERY switch on Shape that needs updating.

static String describe(Object obj) {
    return switch (obj) {
        case null         -> "nothing";
        case Integer i when i < 0 -> "negative int: " + i;
        case Integer i    -> "int: " + i;
        case String s     -> "string of length " + s.length();

Chapter 31 — Virtual Threads and What's Next

Virtual threads are the headline feature of Java 21 and possibly the biggest change to the platform in a decade. This final chapter covers them, structured concurrency, scoped values, and a few features still cooking — so you know what's coming as you continue your Java journey beyond this book.

The problem virtual threads solve

Classic Java threads are one-to-one with operating system threads. They're heavy — a few KB to a few MB of stack each — so most servers cap them around a few thousand. Concurrent server design therefore needed thread pools, async APIs, callbacks, reactive streams... lots of complexity to avoid blocking a precious thread.

Virtual threads change the equation. They are managed by the JVM, not the OS. Millions can exist at once. When one blocks on I/O, the JVM parks it cheaply and uses the underlying OS thread (the carrier) for something else. Write simple, thread-per-request code — and have it scale.

Creating a virtual thread

// Single virtual thread:
Thread.startVirtualThread(() -> {
    System.out.println("Hi from a virtual thread");
});

// Executor backed by virtual threads — one VT per task:
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 10_000; i++) {
        int id = i;
        executor.submit(() -> {
            Thread.sleep(Duration.ofSeconds(1));
            System.out.println("task " + id + " done");
            return null;
        });
    }
}
// 10,000 concurrent virtual threads — totally fine.

Drop-in compatibility

Virtual threads are still java.lang.Thread objects. Code that uses Thread, synchronized, ThreadLocal — it all just works. The difference is how they're scheduled, not the API surface. You don't rewrite your code. You change how the executor is constructed.

When NOT to use virtual threads

  • CPU-bound work. If your tasks just compute (no I/O), virtual threads buy you nothing — you're already limited by cores. Use a fixed pool.
  • Long-running synchronized blocks. A virtual thread holding a synchronized monitor 'pins' its carrier and prevents scheduling. Migrate to ReentrantLock for hot paths.
  • Heavy use of ThreadLocal. Each virtual thread has its own ThreadLocal storage. With millions of threads, this adds up. Scoped values (coming up) are a better fit.

Structured concurrency (preview)

Structured concurrency, in preview since Java 21 and stabilizing in subsequent releases, treats a group of related tasks as a single unit. If one fails, the others cancel. Lifecycle is tied to a scope:

// Preview API — exact syntax may evolve
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    Subtask<User>  user  = scope.fork(() -> fetchUser(id));
    Subtask<Order> order = scope.fork(() -> fetchOrder(id));

    scope.join();             // wait for both
    scope.throwIfFailed();    // propagate any failure

    return new Response(user.get(), order.get());
}

If fetchUser fails, fetchOrder is automatically cancelled. No leaked threads, no manual coordination.

Scoped values (preview)

A modern, immutable replacement for ThreadLocal that works well with virtual threads and structured concurrency:

static final ScopedValue<User> CURRENT_USER = ScopedValue.newInstance();

// Set a value for the duration of a scope:
ScopedValue.where(CURRENT_USER, user).run(() -> {
    // Anywhere inside this block — including child structured scopes —
    // can read CURRENT_USER.get() and see the bound value.
    handleRequest();
});
// Outside the block, the value is gone. Immutable, cheap, safe.

Other coming attractions

  • Foreign Function & Memory API — call native code without JNI; access off-heap memory safely. Stable in Java 22.
  • Stream gatherers — custom intermediate stream operations, like a flexible version of flatMap.
  • Class-file API — read, write, and transform bytecode without ASM or similar.
  • Vector API — SIMD operations for parallel numeric computation.
  • Project Valhalla — value classes that have identity-free, value semantics. Will eventually merge primitives and objects.
  • Project Leyden — faster startup and warmup via ahead-of-time compilation.

Exercise 31.2

Why don't virtual threads help with CPU-bound work?

Solution to Exercise 31.2

Virtual threads' advantage is that they cheaply park when blocked on I/O, freeing the underlying OS thread (carrier) for other work. CPU-bound code never blocks — it's always doing useful work — so there's nothing to park. The number of tasks that can make progress simultaneously is still bounded by the number of CPU cores. For CPU-bound work, a fixed pool sized to your core count is the right tool. Virtual threads shine for I/O-heavy workloads: HTTP servers, database calls, external API calls.
